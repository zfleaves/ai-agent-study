前两课我们用 ollama run 在终端里聊天。但真正的 Agent 开发需要在代码里调用模型——构建工具链、控制流程、测量性能。

🔑 前端直觉

ollama run 就像在浏览器里手动敲 URL 测试 API。到了真正开发，你得用 fetch() / axios（而 ollama-js 就是 Ollama 的专用 SDK）。

二、知识：Streaming 原理

非流式（Non-streaming）

客户端                      Ollama
  │                           │
  │──── 发送 Prompt ─────────→│
  │                           │ 生成全部回复（等……）
  │←─── 完整回复 ────────────│ 一次性返回
  │                           │
  总时间 = 全部 token 生成完才返回

流式（Streaming）

客户端                      Ollama
  │                           │
  │──── 发送 Prompt ─────────→│
  │←─── token 1 ─────────────│ 立刻开始返回
  │←─── token 2 ─────────────│
  │←─── token 3 ─────────────│
  │←─── ... ─────────────────│
  │←─── token N ─────────────│ 最后一个 token
  │                           │
  首 token 延迟 < 1s，之后逐个 token 到达
🔑 两个关键指标
指标中文含义
TTFT首 token 延迟从发送 Prompt 到收到第一个 token 的时间
Total Time总延迟从发送 Prompt 到收到最后一个 token 的时间

非流式：TTFT ≈ Total Time（等全部生成完才返回）
流式：TTFT << Total Time（立刻开始返回，逐个 token 到达）

三、知识：ollama-js API

安装

npm init -y
npm install ollama          # 内置 TypeScript 类型，无需 @types/xxx
npm install -D typescript tsx @types/node

非流式调用

import ollama from 'ollama';

const response = await ollama.chat({
  model: 'qwen2.5:3b',
  messages: [{ role: 'user', content: '你好，用一句话介绍自己' }],
});
// response.message.content 是完整回复字符串
console.log(response.message.content);

流式调用

import ollama from 'ollama';

const stream = await ollama.chat({
  model: 'qwen2.5:3b',
  messages: [{ role: 'user', content: '写一首关于编程的五言绝句' }],
  stream: true,  // ← 关键：开启流式
});

// stream 是 AsyncGenerator，用 for await 逐个接收 token
for await (const chunk of stream) {
  process.stdout.write(chunk.message.content);  // 逐字打印
}
console.log(); // 最后换行
⚠️ ollama-js 的 streaming 是 SSE 而非真正的 chunked

ollama-js 的 stream: true 底层使用 Server-Sent Events (SSE)。每个 chunk 可能包含多个 token，不是严格的"一个 token 一个 chunk"。但用户体验上仍然是"逐字蹦出"的效果。

✍️ Step 2：写流式对话脚本

创建 demos/week-02/stream-chat.ts：

import ollama from 'ollama';

const MODEL = 'qwen2.5:3b';
const PROMPT = '用 TypeScript 写一个冒泡排序函数，带类型定义';

async function streamChat() {
  const startTime = Date.now();
  let firstTokenTime = 0;
  let totalTokens = 0;

  const stream = await ollama.chat({
    model: MODEL,
    messages: [{ role: 'user', content: PROMPT }],
    stream: true,
  });

  for await (const chunk of stream) {
    if (firstTokenTime === 0) {
      firstTokenTime = Date.now();
    }
    totalTokens++;
    process.stdout.write(chunk.message.content);
  }

  const endTime = Date.now();
  const ttft = firstTokenTime - startTime;
  const total = endTime - startTime;

  console.log(`\n\n--- 性能报告 ---`);
  console.log(`模型: ${MODEL}`);
  console.log(`首 token 延迟 (TTFT): ${ttft}ms`);
  console.log(`总耗时: ${total}ms`);
  console.log(`生成 token 数: ${totalTokens} chunks`);
  console.log(`Decode 耗时: ${total - ttft}ms`);
}

streamChat().catch(console.error);

六、自测

用纸笔回答，不看答案。

  1. Streaming 和非 Streaming 的核心区别是什么？用户感知上有什么不同？
  2. TTFT 是什么？为什么流式场景下 TTFT 远小于 Total Time？
  3. ollama-js 的 stream: true 返回的是什么类型？怎么遍历它？
  4. 为什么 Agent 开发需要用代码调用模型，而不能只用 ollama run？
  5. 如果用 stream: true 但不用 for await，会发生什么？
参考答案
  1. 非 Streaming：模型生成完所有 token 后才一次性返回完整回复，用户感知是"等半天，突然全出来了"。Streaming：模型生成一个 token 就立刻返回一个 token，用户感知是"逐字蹦出，立即有反馈"。Streaming 的首 token 延迟远小于总延迟，用户体验更好——就像网页的 SSR vs CSR，用户更在意"多快能看到东西"而不是"多快全部加载完"。
  2. TTFT（Time To First Token） 是从发送 Prompt 到收到第一个 token 的时间。在流式场景下，模型不需要等全部 token 生成完才返回——生成第一个 token 就立刻 push 给客户端，所以 TTFT 通常 < 1s。而 Total Time 要等所有 token 生成完，取决于输出长度。TTFT 决定了用户感知的"响应速度"，是流式体验的核心指标。
  3. stream: true 返回的是 AsyncGenerator（异步生成器）。用 for await (const chunk of stream) 遍历，每个 chunk 包含 chunk.message.content（当前片段的文本）。每次迭代 await 等待下一个 chunk 到达，所以是"边生成边消费"的模式。
  4. 因为 Agent 开发需要：① 精确控制工具调用逻辑（模型输出 function_call → 执行 → 返回结果 → 继续推理），命令行无法做这个循环；② 集成到应用（React 前端、API 服务、自动化脚本）；③ 精确测量性能（TTFT、tok/s、内存占用）；④ 类型安全（TypeScript 定义消息格式、工具参数 Schema）。ollama run 只适合手动测试和快速验证。
  5. 如果 stream: true 但用非流式的方式消费（比如直接 await ollama.chat(...) 然后读 response.message.content），你会只拿到一个空字符串或第一个 chunk 的内容。因为流式 API 的设计是"边生成边消费"——你不消费 AsyncGenerator，它就不会积累完整回复。正确的做法是用 for await 逐个消费并拼接。
