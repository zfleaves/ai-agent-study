一、知识：推理的三个阶段

一次推理分三个阶段。Ollama API 返回的指标一一对应：

load_duration:        60.9s  ← 阶段①：模型加载到内存（冷启动）
prompt_eval_duration:  0.7s  ← 阶段②：理解 Prompt（Prefill 预填充）
eval_duration:        59.0s  ← 阶段③：逐 token 生成回复（Decode）
eval rate:            12.9   ← 每秒生成 token 数，核心速度指标
阶段耗时用户感知前端类比
① Load30-60s"怎么还没反应"npm start 冷启动
② Prefill< 1s基本无感Webpack 读入口
③ Decode逐 token"一个字一个字蹦"HMR 热更新
⚠️ 模型只在冷启动时加载

Ollama 默认在模型空闲 5 分钟后从内存卸载。连续对话只有第一次等加载；隔 5 分钟再问，又要重新加载。

二、知识：System Prompt — 模型的人设

System Prompt 是注入在所有对话之前的一段指令，定义"你是谁"和"你怎么说话"。它只出现一次，但影响贯穿整个对话。

┌──────────────────────────────┐
│  System: "你是一个四川话AI"    │  ← 角色设定（只设一次）
├──────────────────────────────┤
│  User: "你好"                 │  ← 用户消息
│  Assistant: "你好噻！"        │  ← 模型回复（受 System 影响）
│  User: "今天天气咋样"          │
│  Assistant: "巴适得板！"      │
└──────────────────────────────┘
🧪 实验：System Prompt

无 System Prompt：

ollama run qwen2.5:3b
>>> 你是谁？

有 System Prompt：

ollama run qwen2.5:3b
>>> /set system 你是一个只讲四川话的AI助手，所有回答必须用四川话
>>> 你是谁？

同一问题，回复天差地别。

三、知识：Temperature — 创造力的旋钮

Temperature 控制模型输出的随机性。模型生成每个 token 时计算概率分布，Temperature 调整分布的"陡峭程度"：

Temperature效果适用场景
0.0每次输出完全一样，最保守代码生成、数学计算
0.3-0.5稳定但略有变化翻译、总结
0.7-0.9有创意但不离谱日常对话、内容创作
1.0+天马行空，可能胡说创意写作、头脑风暴
🧪 实验：Temperature
ollama run qwen2.5:3b
>>> /set parameter temperature 0.0
>>> 给我 3 个 TypeScript 变量名，用来存用户信息
  （记下结果）
>>> /set parameter temperature 1.5
>>> 给我 3 个 TypeScript 变量名，用来存用户信息
  （对比 — 高温度下是不是更"放飞"了？）

四、动手：速度实测

📊 任务：对比两个模型的 tok/s

用同一个 Prompt，分别测试两个模型，记录 eval rate：

ollama run qwen2.5:3b --verbose
>>> 用 TypeScript 写一个冒泡排序函数，带类型定义

ollama run hermes3:8b --verbose
>>> 用 TypeScript 写一个冒泡排序函数，带类型定义

每次回复后 --verbose 会打印：

eval count:     256 token(s)     ← 生成了多少 token
eval rate:      17.5 tokens/s   ← 每秒生成速度 ← 记这个！

把你两个模型的 eval rate 记下来，这就是你的速度实测报告。

五、命令速查

ollama run <model> --verbose       # 对话 + 性能数据
ollama run <model> "一句话"         # 单次对话（非交互）

# 交互模式命令（>>> 提示符后）：
/set system <prompt>               # 设置 System Prompt
/set parameter temperature <值>    # 设置 Temperature（0.0-2.0）
/show system                       # 查看当前 System Prompt
/show parameters                   # 查看当前参数
/bye                               # 退出

六、自测

用纸笔回答，不看答案。

  1. 推理三阶段分别叫什么？哪个最耗时？
  2. System Prompt 和 User Prompt 区别？
  3. Temperature 0.0 和 1.5 输出差在哪？
  4. 代码生成为什么用低 Temperature？
  5. eval rate 衡量什么？单位是什么？
参考答案
  1. Load（模型加载到内存，30-60s，只在冷启动时发生）、Prefill（理解 Prompt，< 1s，基本无感）、Decode（逐 token 生成回复，取决于输出长度）。Decode 最耗时，用户感知到的"一个字一个字蹦"就是 Decode 阶段，衡量指标是 eval rate（tok/s）。
  2. System Prompt 是"角色设定"，在所有对话之前注入，只设一次，定义"你是谁"和"你怎么说话"；User Prompt 是用户每次输入的具体问题，每次对话都有。System Prompt 的影响贯穿整个对话，所以同一句"你是谁？"在不同 System Prompt 下回复完全不同。
  3. Temperature 控制模型输出时选择 token 的随机性。0.0：只选概率最高的 token，每次输出完全一样，适合代码生成和事实问答；1.5：概率分布接近均匀，几乎随机选 token，输出有天马行空的创意但也可能胡说八道，适合头脑风暴。
  4. 代码生成要求精确、可复现——同一个输入应该产生同一个输出。低 Temperature 确保模型每次都选最高概率的 token，不会"发挥创意"把语法写错或用不存在的 API。如果让 Temperature=1.5 的模型写代码，它可能"发明"一个不存在的函数名。
  5. eval rate 衡量模型在 Decode 阶段生成回复的速度，单位是 tokens/s（每秒生成多少 token）。这是最核心的速度指标——值越高，回复越快。通过 --verbose 参数可以看到每次推理的 eval rate，用来对比不同模型在你电脑上的实际表现。
