🔑 关键设计决策
  • messages 数组：每次对话后 push user + assistant 消息，这是多轮对话的基础。模型不"记住"历史——是你把历史每次都发给它。
  • stats 数组：每轮记录 TTFT、总耗时、输出长度。这些数据让你在选模型时不是"凭感觉"，而是"看数据"。
  • 默认用 3B：交互式聊天优先级是"响应快"而非"质量最高"。3B 的 200ms TTFT 比 8B 的 1000ms 更适合聊天。

四、本周常见踩坑总结

⚠️ 第 2 周踩坑记录
  1. Ollama 没启动：运行脚本前确认 Ollama 在后台运行（任务栏有图标或 ollama list 能返回模型列表）。
  2. 模型没拉取：ollama pull hermes3:8b 需要先下载模型，8B 模型约 4.7GB，首次下载需 10-20 分钟。
  3. Node.js 版本太低：ollama-js 需要 Node.js 18+。用 node --version 检查。
  4. tsx 没安装：npx tsx 首次运行会自动下载。也可以全局安装：npm i -g tsx。
  5. 对话历史太长：多轮对话后 messages 数组越来越大，超出 Context Window 会导致模型"忘记"前面的内容。用 /clear 定期清理。
  6. 8B 模型太慢：CPU 推理 8B 模型首 token 可能等 1-2 秒。这是正常的——不是你的代码有问题，是模型太大。开发时用 3B，需要质量时切 8B。

📝 参考答案

第 1 题：为什么 Ollama 使用 GGUF 格式？

核心原因：原始 PyTorch 模型文件（.pt / .safetensors）是 16 位浮点精度（FP16），一个 8B 模型需要约 16GB 显存/内存。普通开发者的电脑根本跑不动。GGUF 格式做了两件事：

  1. 量化压缩：把 FP16 参数压缩到 4-bit 整数（Q4_K_M），模型大小从 16GB → 4.7GB，缩小约 3.4 倍。而且 Q4 量化几乎不损失质量——因为模型参数本身就不需要 16 位精度。
  2. 自包含：GGUF 文件包含模型权重 + 分词器 + 元数据（模型架构、量化参数、特殊 token 定义），一个文件就是完整的模型。不需要 Python 环境、不需要 PyTorch、不需要 transformers 库。

为什么量化几乎不损失质量？ 神经网络在训练时学到的参数分布是连续的，但推理时不需要那么高的精度。就像一张 4K 照片缩小到 1080p——你仍然能认出里面的人。Q4 量化用 16 个离散值（4-bit）来近似连续的浮点值，误差在 1-2% 以内，对文本生成质量的影响微乎其微。

第 2 题：Streaming 和非 Streaming 的 TTFT 有什么区别？

答案：TTFT 本身是一样的，但用户感知完全不同。

TTFT（首 token 延迟）主要由 Prefill 阶段决定——模型需要先"阅读"整个 Prompt 并计算所有 token 之间的注意力关系。这个计算量取决于 Prompt 长度，跟是否 Streaming 无关。

关键区别在于感知延迟：

  • 非 Streaming：用户要等模型生成完所有 token 后才能看到任何内容。如果生成 500 个 token，每个 token 40ms，用户要等 20000ms（20 秒）才能看到结果。
  • Streaming：模型生成第一个 token 就立即返回。用户看到 TTFT（比如 400ms）+ 每 40ms 一个新 token。虽然总时间一样，但用户感觉"它正在打"而不是"卡住了"。

类比：非 Streaming 像等一封完整的邮件；Streaming 像看着对方打字——后者让你知道"对方在回复"，心理等待感更短。

第 3 题：三种策略如何选择？

判断标准：问自己三个问题——

  1. "这个任务模型在训练中见过吗？" → 见过 → Zero-shot 就够了。没见过（如公司内部 DSL）→ 需要 Few-shot。
  2. "模型的困难是'不知道格式'还是'不知道答案'？" → 格式问题 → Few-shot（1-2 个例子）。知识问题 → Zero-shot + 清晰指令。
  3. "这个任务的核心挑战是'知道答案'还是'推导答案'？" → 推导答案 → 需要 CoT。
场景 策略 原因
写一个冒泡排序（TS） Zero-shot 模型预训练中见过无数次
把自然语言转成 SQL Few-shot 格式控制是核心，1-2 个例子足够
数学应用题（"水池注水"） CoT 需要多步推理，不是一步知道答案
创意写作（"写一首诗"） Zero-shot CoT 会破坏流畅性，Few-shot 限制创意

第 4 题：为什么"刚好够用"而不是"尽量短"？

因为"尽量短"可能导致多轮纠正，总时间反而更长。

假设你用极简 Prompt "写个排序"：

  • 第 1 轮：模型输出 Python 版（不对）→ TTFT 200ms + 生成 2000ms = 2200ms
  • 第 2 轮：你说"要 TypeScript"→ 模型输出 TS 但没类型 → 2200ms
  • 第 3 轮：你说"加类型定义"→ 终于对了 → 2200ms
  • 总耗时：6600ms

如果用完整 Prompt "用 TypeScript 写一个冒泡排序函数，输入 number[]，返回排序后的数组，带完整类型定义"：

  • 第 1 轮：TTFT 400ms + 生成 2000ms = 2400ms
  • 总耗时：2400ms

完整 Prompt 反而快了近 3 倍。"刚好够用"的意思是：保留所有必要信息（任务 + 格式 + 约束），去掉冗余（礼貌用语、重复描述）。

判断标准：如果再删掉任何一个词，输出质量就会明显下降——这个临界点就是最优长度。

第 5 题：messages 数组的作用和风险

作用：messages 数组是对话历史的存储。每次调用 ollama.chat({ messages }) 时，你把整个对话历史发给模型。模型本身不保存任何状态——它每次都从零开始"阅读"你给的 messages。多轮对话的"记忆"不是模型的能力，而是你把历史每次都发给它的结果。

如果不清空：

  1. Context Window 溢出：qwen2.5:3b 的上下文窗口是 32768 tokens。每轮对话的 user + assistant 消息加起来可能几百 token。如果一直不清理，几十轮后 messages 的 token 数超过 32768——模型会截断最早的消息，导致"忘记"对话开头的内容。
  2. TTFT 越来越慢：每次 Prefill 要处理整个 messages 数组。messages 越长，Prefill 越久，用户感觉回复越来越慢。
  3. token 浪费：旧的对话内容对当前问题可能毫无帮助，但仍然在消耗 token 预算。

解决方案：Demo 1 中提供了 /clear 命令手动清空。在真正的 Agent 应用中，需要更智能的策略——滑动窗口（保留最近 N 轮）、摘要压缩（把旧对话总结成一段话）、或者 RAG（检索相关历史而非全部保留）。这些会在后续课程中深入学习。
