一、从"单次调用"到"循环"

前两节课我们看到的 Tool-calling 是"一次性的"：用户提问 → 模型调工具 → 执行 → 回答。但真正的 Agent 需要多轮决策——一个任务可能需要多次工具调用、中间结果需要验证、出错了需要重试。

1.1 为什么需要循环？

📋 场景：用户问"北京和上海哪个更热？"
  1. 模型不知道北京和上海的天气 → 需要调 getWeather("北京")
  2. 拿到北京天气，但还需要上海天气 → 再调 getWeather("上海")
  3. 拿到两个城市天气，可以比较了 → 生成最终回答

这个任务需要 2 次工具调用。如果不循环，模型第一次调完工具就停了，任务完不成。

1.2 前端直觉类比

// 普通调用 = 一次 fetch
const res = await fetch('/api/chat', { body: question });
// → 拿到回答，结束

// Agent 循环 = 带重试的异步流水线
let done = false;
while (!done && steps < maxSteps) {
  const decision = await llm.decide(messages);
  if (decision.wantsToCallTool) {
    const result = await executeTool(decision.toolCall);  // 执行工具
    messages.push({ role: 'tool', content: result });      // 回传结果
    // 继续循环，让模型再决策
  } else {
    done = true;  // 模型认为任务完成
  }
}

二、Agent 循环的 5 个关键设计

2.1 循环结构

┌─────────────────────────────────────────────────────┐
│                                                     │
│   while (step < MAX_STEPS) {                        │
│                                                     │
│     ① response = await llm.chat(messages, tools)    │
│                                                     │
│     ② if (没有 tool_calls) → 任务完成，退出循环      │
│                                                     │
│     ③ for (toolCall of tool_calls) {                │
│         result = executeTool(name, args)  // 执行    │
│         messages.push({ role: "tool", result })      │
│       }                                              │
│                                                     │
│     ④ 回到 ①，模型基于新信息继续决策                  │
│   }                                                  │
│                                                     │
└─────────────────────────────────────────────────────┘

2.2 终止条件：什么时候停？

循环不能永远跑下去。你需要两个终止条件：

条件 触发时机 代码
自然终止 模型不再调用工具，直接生成文本回答 if (!toolCalls) break;
强制终止 达到最大步数（如 5 步），防止无限循环 while (step < MAX_STEPS)
⚠️ MAX_STEPS 设多少合适？
  • 太少了（如 2）：复杂任务完不成，用户体验差
  • 太多了（如 20）：模型可能陷入循环，浪费 token 和时间
  • 推荐值：5-8。大多数任务在 3-5 步内完成，超过 8 步说明任务需要重新设计

2.3 链式调用 vs 并行调用

模型可能一次返回多个 tool_call，你需要决定怎么处理：

🔗 串行（链式）

// 工具 A 的结果是工具 B 的输入
getWeather("北京") → 25°C
getWeather("上海") → 28°C
// 再比较两个结果

适用：结果有依赖关系

⚡ 并行

// 两个工具互不依赖，同时执行
Promise.all([
  getWeather("北京"),
  getWeather("上海"),
])
// 拿到两个结果，一起回传

适用：结果互不依赖，节省时间

判断标准：问自己"工具 B 需要工具 A 的结果作为参数吗？"——需要 → 串行；不需要 → 并行。

2.4 错误处理：工具失败了怎么办？

工具执行可能失败：网络超时、API 限流、参数格式错误……Agent 必须有容错能力：

// ❌ 坏：异常直接抛出，循环崩溃
const result = executeTool(name, args);

// ✅ 好：捕获异常，作为正常结果回传，让模型决定下一步
try {
  const result = executeTool(name, args);
  messages.push({ role: "tool", content: result });
} catch (err) {
  messages.push({
    role: "tool",
    content: `工具执行失败: ${err.message}`,
  });
  // 模型收到错误后可以：
  //   ① 换一种方式调用（如换个参数）
  //   ② 换一个工具
  //   ③ 如实告知用户
}
💡 关键设计：把错误也当作"正常结果"

不要把异常抛出循环——把错误信息以 role: "tool" 回传给模型。模型会自己判断：换个参数重试？换另一个工具？还是告诉用户"抱歉，做不到"？这是 Agent 的"韧性"来源。

2.5 消息历史管理

循环中 messages 数组会快速增长。每次循环添加 assistant 消息 + tool 结果，容易超出 Context Window：

// 一个典型的 3 步循环后的 messages 结构：
[
  { role: "system",  content: "..." },           // System Prompt
  { role: "user",    content: "查天气+计算" },     // 用户输入
  { role: "assistant", content: "", tool_calls }, // 第 1 步：调 getWeather
  { role: "tool",    content: "北京 25°C" },      // 工具结果
  { role: "assistant", content: "", tool_calls }, // 第 2 步：调 calculate
  { role: "tool",    content: "25×365=9125" },    // 工具结果
  { role: "assistant", content: "最终回答..." },   // 第 3 步：最终回答
]

对于本地小模型（3B-8B，Context 32k），这个量级完全没问题。但后续学到更复杂的 Agent 时，需要引入滑动窗口和摘要压缩策略。

三、技能：完整的 Agent 循环代码

以下是生产级 Agent 循环的骨架代码（也是 Demo 3 的核心）：

async function agentLoop(userInput: string): Promise<string> {
  const messages: Message[] = [
    { role: 'system', content: SYSTEM_PROMPT },
    { role: 'user', content: userInput },
  ];

  let step = 0;
  const MAX_STEPS = 5;

  while (step < MAX_STEPS) {
    step++;

    // ① 调用模型
    const response = await ollama.chat({
      model: MODEL,
      messages,
      tools: TOOLS,
    });

    const toolCalls = response.message.tool_calls;

    // ② 终止条件：模型不再调工具
    if (!toolCalls || toolCalls.length === 0) {
      return response.message.content;  // ← 任务完成
    }

    // ③ 保存 assistant 消息
    messages.push(response.message);

    // ④ 执行工具 + 回传结果
    for (const tc of toolCalls) {
      try {
        const result = executeTool(tc.function.name, tc.function.arguments);
        messages.push({
          role: 'tool',
          content: result,
          tool_call_id: tc.id,
        });
      } catch (err) {
        messages.push({
          role: 'tool',
          content: `错误: ${err.message}`,
          tool_call_id: tc.id,
        });
      }
    }
    // ⑤ 回到循环顶部，模型继续决策
  }

  // ⑥ 达到最大步数，强制结束
  return forceFinalAnswer(messages);
}

四、自测

5 题，覆盖 Agent 循环的核心设计。

  1. 为什么 Agent 需要循环，而不是一次工具调用就结束？
  2. Agent 循环的两个终止条件分别是什么？各自解决什么问题？
  3. 工具执行失败时，为什么应该把错误信息回传给模型而不是抛出异常？
  4. 什么时候应该并行执行多个工具调用，什么时候应该串行？
  5. MAX_STEPS 设得太小和太大各有什么问题？如何确定合理的值？
📝 参考答案

第 1 题：为什么需要循环？

因为现实任务往往需要多次信息获取才能完成，一次工具调用只能拿到一个维度的信息。

举个例子：用户问"北京和上海哪个更热？"

  • 第 1 步：调 getWeather("北京") → 拿到 25°C
  • 第 2 步：还需要上海的数据 → 调 getWeather("上海") → 拿到 28°C
  • 第 3 步：两个数据都有了，可以比较 → 直接回答"上海更热，28°C vs 25°C"

底层原因：模型在单次推理中生成的 tool_calls 是"基于当前已有的信息"做出的决策。它不知道工具结果会是什么——只有在拿到结果后，才能决定下一步。这就像一个人查资料：先查第一本书，发现还缺信息，再查第二本，直到信息够了才写结论。你不能要求他"一次说出所有书的名字"，因为他不确定第一本书里有什么。

判断标准：如果任务只需要一个维度的信息（如"今天北京天气"），一次调用就够了。如果任务需要比较、计算、或基于中间结果做新决策，就一定需要循环。

第 2 题：两个终止条件

① 自然终止：模型不再输出 tool_calls，而是直接生成文本回答。这说明模型认为"信息够了，可以回答了"。

原理：模型在每次收到工具结果后，会重新评估"我还需要更多信息吗？"。当它判断"不需要"时，就会从"工具调用模式"切换到"文本回答模式"。这个切换体现在 token 生成上——模型不再输出 {"function": ... 的 JSON，而是输出自然语言。

② 强制终止：达到 MAX_STEPS 上限。防止模型陷入"永远觉得不够"的循环——比如反复调用同一个工具但拿不到想要的结果，或者任务确实太复杂。

为什么需要强制终止：模型不是完美的。它可能：

  • 陷入"查→不满意→再查→还不满意→再查"的死循环
  • 对简单问题过度拆解，每一步只做一点点
  • 工具返回的错误信息被误解为"需要换一种方式重试"

强制终止是安全网——确保即使模型失控，系统也不会无限消耗 token 和时间。

第 3 题：为什么错误信息要回传而不是抛异常？

因为抛出异常会终止循环，而回传错误信息让模型有机会"自我修复"。

假设用户问"东京天气怎么样"，但你的工具只支持中国城市。如果抛出异常：

  • ❌ 循环崩溃 → 用户看到"系统错误" → 体验很差

如果回传错误信息：

  • ✅ 模型收到 "未找到 东京 的天气数据。支持：北京、上海、深圳、杭州、成都"
  • ✅ 模型可以回复："抱歉，我目前只能查询中国城市的天气。支持的城市有：北京、上海、深圳、杭州、成都。你想查哪个？"
  • ✅ 用户体验好，而且知道了工具的边界

更深层的设计哲学：Agent 的"智能"不在于不出错，而在于出错后能自己处理。把错误暴露给模型（而不是吞掉），模型可以：

  1. 换参数重试：传了 "Beijing" → 错误 → 改成 "北京"
  2. 换工具：getWeather 不支持 → 试试 searchWeb 搜索天气
  3. 降级处理：查不到天气 → 告诉用户"我目前无法获取该城市天气，建议查看天气 App"

这就是 ReAct 模式中 "Reasoning + Acting" 的完整闭环——Action 的结果（包括错误）驱动下一轮 Reasoning。

第 4 题：并行 vs 串行

核心判断：工具 B 的参数是否依赖工具 A 的结果？

场景 策略 原因
查北京和上海的天气 ⚡ 并行 两个查询互不依赖
先查天气，再根据温度决定是否算数 🔗 串行 第 2 步取决于第 1 步的结果
同时查时间、天气、搜索 ⚡ 并行 三个独立操作

实现方式：模型一次返回多个 tool_call 时，如果它们互不依赖，用 Promise.all() 并行执行，所有结果一起回传。如果模型分开两次返回 tool_call，那是模型认为需要串行——你不需要猜测，照模型说的做就行。

性能差异：假设每个工具调用耗时 200ms，2 个互不依赖的工具串行执行 = 400ms，并行执行 = 200ms。在用户体验上，200ms 的差异是"流畅"和"卡顿"的区别。

第 5 题：MAX_STEPS 的取值

太小（如 2）的问题：复杂任务被强制截断。比如"查北京和上海天气，比较哪个更热"需要 3 步——查北京 → 查上海 → 比较回答。如果 MAX_STEPS = 2，第二步查到上海天气后就被截断了，用户得不到最终回答。

太大（如 20）的问题：

  • token 浪费：每步都消耗 token（Prompt 里的 messages 越来越长），20 步可能消耗数千 token 却什么都没完成
  • 延迟累积：每步 500ms × 20 = 10 秒，用户早就关了
  • 模型可能"绕圈"：没有自然终止的任务（如模型不满意义前的结果，反复重试），会一直跑到 MAX_STEPS

如何确定合理值：

  1. 从任务复杂度估算：每个独立的信息获取 ≈ 1 步。查天气+计算 = 2-3 步。查天气+搜索+计算+比较 = 4-5 步。
  2. 观察实际数据：跑 20 个真实任务，统计完成所需步数，取 P95 值 + 1 作为上限。
  3. 分层设置：简单 Agent（< 3 个工具）→ MAX_STEPS = 3。复杂 Agent（5+ 个工具）→ MAX_STEPS = 8。

推荐起步值：5。覆盖绝大多数场景，同时防止无限循环。后续根据实际使用数据调整。

✅ 本课小结

你今天学到了：

  • Agent 循环 = while(未完成) { 模型决策 → 执行工具 → 回传结果 }
  • 两个终止条件：自然终止（模型不再调工具）+ 强制终止（MAX_STEPS）
  • 并行 vs 串行：互不依赖的 tool_call 用 Promise.all() 并行执行
  • 错误处理：把错误信息回传给模型，让它自己决定下一步
  • MAX_STEPS 推荐起步值 5，覆盖大部分场景，防止无限循环
