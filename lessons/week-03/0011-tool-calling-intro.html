一、为什么需要 Tool-calling？

1.1 LLM 的天生局限

前两周我们学到的 LLM 是"纯文本机器"：输入文字 → 输出文字。这在很多场景下够用，但遇到这些问题就傻眼了：

❌ 纯文本 LLM 做不到的事情
  • "现在几点了？" → 模型不知道当前时间，它的训练数据是过去的
  • "北京今天天气怎么样？" → 模型没有实时数据
  • "帮我算一下 1234 × 5678" → 模型是概率预测，不是计算器，大数乘法可能算错
  • "搜索最新的 React 19 文档" → 模型的知识截止于训练日期
  • "把这个文件内容发到 Slack" → 模型没有手，不能操作外部系统

这些问题的共同点是：答案不在模型参数里，而在外部世界。模型需要"伸手"去拿——这就是 Tool-calling。

1.2 前端直觉类比

如果你只写过前端，可以这样理解：

// 普通 LLM  = 纯函数，只能返回字符串
function askLLM(question: string): string {
  return "根据我的训练数据，答案是...";
}

// Tool-calling LLM = 可以调用外部 API 的函数
async function askAgent(question: string): Promise<string> {
  const toolCall = await llm.decide("需要调什么工具？");
  const result = await executeTool(toolCall);  // ← 调用真实的 API
  return await llm.answer("拿到结果了：" + result);
}

类比：普通 LLM 像是一个只能收发短信的秘书——你问什么，他凭记忆回答。Tool-calling Agent 像是秘书有了手机——他可以打电话问天气、查日历、发邮件，然后把结果汇报给你。

二、核心机制：它是怎么工作的？

2.1 三步循环

Tool-calling 的本质是一个请求→决策→执行→反馈的循环：

┌─────────────────────────────────────────────────────┐
│                                                       │
│   ① User: "北京今天天气怎么样？"                         │
│      → 发送给模型（带上可用工具列表）                      │
│                                                       │
│   ② LLM: 我不会直接知道天气，但有个 getWeather 工具       │
│      → 输出结构化 JSON：                                │
│      { "function": "getWeather", "args": { "city": "北京" } } │
│                                                       │
│   ③ 你的代码: 执行 getWeather("北京") → 拿到结果          │
│      → 把结果追加到对话中："北京今天晴，25°C"              │
│                                                       │
│   ④ LLM: 收到天气结果，生成最终回答                       │
│      → "北京今天晴天，气温 25°C，适合出门。"              │
│                                                       │
└─────────────────────────────────────────────────────┘

关键洞察：模型不是"执行"工具——它只是输出一个 JSON 告诉你"我想调用这个函数"。你的代码才是真正执行工具的人。模型是大脑，你的代码是手。

2.2 请求格式：如何告诉模型"你能用什么工具"？

在发送消息时，除了 messages 数组，还需要附加一个 tools 数组——用 JSON Schema 描述每个工具：

// 工具定义：用 JSON Schema 描述函数的签名和用途
const tools = [
  {
    type: "function",
    function: {
      name: "getWeather",           // 函数名
      description: "获取指定城市的当前天气信息",  // 用途描述（关键！）
      parameters: {                  // 参数定义（JSON Schema）
        type: "object",
        properties: {
          city: {
            type: "string",
            description: "城市名称，如 '北京'、'上海'",
          },
        },
        required: ["city"],          // 必填参数
      },
    },
  },
];

// 发送请求时带上 tools
const response = await ollama.chat({
  model: "hermes3:8b",
  messages: [{ role: "user", content: "北京今天天气怎么样？" }],
  tools,  // ← 告诉模型：你可以用这些工具
});
💡 description 字段比你想象的更重要

模型选择哪个工具、传递什么参数，全靠 description 来判断。一个模糊的 description 会导致模型选错工具或传错参数。写 description 就像写 API 文档——要精确、具体、带例子。

2.3 响应格式：模型如何"说"它想调用工具？

当模型决定调用工具时，它不会输出普通文本，而是在 message 对象中附带 tool_calls 数组：

// 模型返回的响应（简化版）
{
  message: {
    role: "assistant",
    content: "",                    // ← 工具调用时 content 通常为空
    tool_calls: [
      {
        id: "call_abc123",         // 调用 ID，用于后续关联结果
        type: "function",
        function: {
          name: "getWeather",      // 要调用的函数名
          arguments: {             // 参数（JSON 对象）
            city: "北京"
          },
        },
      },
    ],
  },
}

你的代码解析这个响应 → 执行 getWeather("北京") → 把结果告诉模型 → 模型生成最终回答。

2.4 工具结果回传：如何让模型"知道"结果？

执行完工具后，你需要把结果以 tool 角色 追加到消息历史中：

// 把工具执行结果追加到 messages
messages.push({
  role: "tool",                    // ← 特殊角色：tool
  content: JSON.stringify({        // 工具返回的结果
    city: "北京",
    temperature: 25,
    condition: "晴天",
    humidity: "45%",
  }),
  tool_call_id: "call_abc123",    // 对应之前的调用 ID
});

// 再次调用模型，它现在能基于结果回答了
const finalResponse = await ollama.chat({
  model: "hermes3:8b",
  messages,
  tools,
});

三、Hermes 3 的 Tool-calling 特性

3.1 Hermes 的独特之处

Hermes 3 与其他模型在 Tool-calling 上的关键差异：

特性 通用模型（如 GPT-4） Hermes 3
训练方式 RLHF + 系统提示 DPO + 合成数据微调
工具调用格式 OpenAI 标准格式 兼容 OpenAI 格式 + 支持 XML 格式
并行调用 支持 支持（一次返回多个 tool_call）
工具数量 官方建议 ≤ 128 实测 5-10 个工具效果最佳
中文工具调用 GPT-4 优秀 推荐用 Qwen2.5（中文更好）
🔑 Hermes 3 的 Tool-calling 格式

Hermes 3 支持两种 tool-calling 格式：

  1. OpenAI 兼容格式（推荐）：和上面的 tools 数组、tool_calls 格式完全一致。Ollama 的 ollama.chat() 原生支持这种格式。
  2. XML 格式：用 <tool_call> XML 标签包裹。这是 Hermes 训练数据中使用的格式，但 Ollama 的 JS SDK 已经帮我们处理了转换。

3.2 tool_choice 参数：控制模型的"调用意愿"

tool_choice 参数告诉模型是否必须调用工具：

值 含义 适用场景
"auto" 模型自己决定是否调用工具（默认） 通用场景——模型会判断"这个问题需要工具吗？"
"required" 强制模型必须调用某个工具 Agent 循环中，你知道当前步骤必须用工具
"none" 禁止模型调用工具 你只想让模型做纯文本回复
{"type": "function", "function": {"name": "getWeather"}} 强制调用指定工具 路由场景——你知道用户意图就是查天气

五、动手：理解 JSON Schema 工具定义

✏️ 练习：设计一个工具定义

假设你要让 Agent 能搜索代码库。请用 JSON Schema 写出这个工具的定义：

  • 函数名：searchCodebase
  • 功能：在项目代码中搜索指定关键词
  • 参数：query（搜索关键词，必填）、fileType（文件类型，可选，如 .ts、.tsx）
📝 参考答案 
{
  type: "function",
  function: {
    name: "searchCodebase",
    description: "在项目代码中搜索指定关键词。返回匹配的文件路径和行号。例如搜索 'useState' 找到所有使用该 Hook 的组件。",
    parameters: {
      type: "object",
      properties: {
        query: {
          type: "string",
          description: "要搜索的关键词，支持正则表达式，如 'useEffect'、'import.*from'",
        },
        fileType: {
          type: "string",
          description: "限定文件扩展名，如 '.ts'、'.tsx'、'.json'。不传则搜索所有文件。",
        },
      },
      required: ["query"],
    },
  },
}

要点：

  • description 要具体——"搜索代码"不够，要说清楚"返回什么"、"搜索什么"
  • parameters.description 要带例子——模型看了例子才知道怎么传参数
  • required 只标记真正必填的——fileType 不在 required 里，说明是可选的

📝 参考答案

第 1 题：模型真的"执行"了工具吗？

没有。模型只是输出一个结构化的 JSON，告诉你"我想调用这个函数，参数是这些"。

这是理解 Tool-calling 最关键的认知：模型是纯粹的文本生成器。它不能执行代码、不能访问网络、不能操作文件系统。它唯一的能力是预测下一个 token。Tool-calling 的本质是：模型在训练中学会了"当我需要外部信息时，我应该输出一个特殊的 JSON 格式"。

底层的自回归机制：模型在生成每个 token 时，都是基于之前所有 token 的条件概率分布。当它看到系统提示 + 工具定义 + 用户问题后，概率最高的下一个 token 可能是 {（开始 JSON），然后是 "、function、"……而不是一个自然语言回答。这不是模型"决定"调用工具——而是训练数据让它学会了在这种上下文中输出工具调用 JSON 的概率最高。

类比：模型像一个只会写字的机器人，它的手被绑在键盘上。它不能自己站起来去拿杯子，但它可以写下一张纸条："请帮我拿杯水，杯子在厨房桌上"。你的代码就是那个读纸条并执行的人。

第 2 题：为什么 description 比 name 更重要？

因为模型是"读"description 来理解工具用途的，而不是"记"name。

模型在预训练和微调中没有见过你的工具。它不知道 getWeather 这个函数名是什么意思。它只能通过你写的 description 来推断：这个工具是干嘛的？什么时候该用它？参数怎么填？

具体例子：

Description 质量 内容 模型行为
❌ 差 "获取天气" 可能传错参数，比如传 "Beijing" 而不是 "北京"
⚠️ 一般 "获取指定城市的天气" 知道要传城市名，但不知道格式
✅ 好 "获取指定城市的当前天气信息，返回温度、湿度、天气状况。城市名用中文，如 '北京'、'上海'。" 正确传参，格式匹配

判断标准：写完 description 后问自己——"如果我是第一次看到这个工具的人，我能不能准确知道它做什么、怎么用？"如果答案是"不能"，那就需要更详细。

第 3 题：auto vs required 的区别和适用场景

核心区别：auto 让模型自己判断"这个问题需要工具吗？"，required 强制模型"必须调用工具，不能直接回答"。

底层机制：当 tool_choice: "required" 时，模型在生成时的 token 空间被约束——必须以工具调用 JSON 开始，而不能以自然语言开始。这相当于在采样的第一步就限制了可选 token 的范围。当 tool_choice: "auto" 时，模型先从完整的 token 空间中采样，如果它觉得"回答"比"调工具"更合适，就会输出自然语言。

场景 应该用 原因
自由聊天（用户可能问各种问题） auto 大部分问题不需要工具，强制调用反而糟糕
Agent 循环的第一步（必须分析用户意图） required 确保 Agent 一定会调用工具，不会"偷懒"直接回答
用户明确要求"帮我查..." required 或指定工具 用户意图明确，不需要模型再判断
工具执行完成后，让模型总结结果 none 或 auto 此时目标是生成回答，不是再调工具

判断标准：问自己——"在当前步骤，模型还有没有'不调工具直接回答'的正确可能？" 如果有 → auto；如果没有 → required。

第 4 题：工具结果回传时 role 为什么是 "tool"？

因为模型需要区分"这是工具执行的结果"和"这是用户/助手说的话"。

在 Transformer 的注意力机制中，每条消息的 role 标签会影响模型对这条消息的"态度"：

  • user 角色：模型认为这是"需要回答的问题"，会认真对待并尝试生成回复
  • assistant 角色：模型认为这是"自己之前说的话"，用于保持对话连贯性
  • tool 角色：模型认为这是"工具返回的数据"，是客观事实，不需要质疑——直接基于这些数据生成回答

如果错误地用 user 角色传工具结果，模型可能会把结果当作"用户的新问题"来理解，导致混淆。如果用 assistant 角色，模型可能会认为"这是我自己说的"——但实际上工具结果不是模型生成的。

类比：role 就像消息的"发件人标签"。user = 老板的指令，assistant = 你自己的笔记，tool = 外部数据源返回的报告。你把报告贴在自己笔记里，老板看不到，也不知道有这个数据。

第 5 题：Hermes 3 两种格式应该用哪种？

使用 OpenAI 兼容格式（JSON 格式）。

原因：

  1. Ollama 的 JS SDK 原生支持：ollama.chat() 的 tools 参数就是 OpenAI 格式，SDK 内部会自动处理与 Hermes 的兼容。你不需要手写 XML 标签。
  2. 生态兼容性：OpenAI 格式是行业标准。Vercel AI SDK、LangChain、Spring AI 等框架都支持这种格式。用标准格式意味着你的代码可以在不同模型之间切换，不需要改工具定义。
  3. XML 格式是训练格式，不是接口格式：Hermes 训练数据中用 XML 标签标注工具调用，但推理时 Ollama 已经帮我们做了格式转换。你不需要关心底层。

唯一例外：如果你在用 ollama.generate()（非 chat API）直接做文本补全，模型可能输出 XML 格式的工具调用。此时需要手动解析 <tool_call> 标签。但课程中我们统一使用 ollama.chat() + OpenAI 格式，不需要处理这种情况。

✅ 本课小结

你今天学到了：

  • Tool-calling 是 LLM 从"会说话"到"会做事"的关键能力
  • 模型不执行工具——它输出 JSON 告诉你的代码"我想调用这个函数"
  • 工具定义的核心是 JSON Schema 的 description——写得越好，模型选得越准
  • tool_choice 控制模型的调用意愿：auto / required / none / 指定工具
  • 工具结果用 role: "tool" 回传，让模型区分"数据"和"对话"
  • Hermes 3 推荐使用 OpenAI 兼容格式，Ollama SDK 原生支持
