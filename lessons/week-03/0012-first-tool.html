一、回顾：工具定义长什么样？

上节课我们知道了工具定义是一个 JSON Schema 对象。这节课深入：怎么写出好的工具定义。

// 一个完整的工具定义 = 3 个关键部分
{
  type: "function",           // ① 固定值
  function: {
    name: "getWeather",       // ② 函数名（模型用它来标识）
    description: "...",       // ③ 描述（模型用它来理解）← 最重要！
    parameters: {             // ④ 参数 Schema（模型用它来填参）
      type: "object",
      properties: { ... },
      required: [...],
    },
  },
}
🧠 模型怎么"读"工具定义？

模型不是代码编译器——它不会"理解"你的函数签名。工具定义是自然语言指令的一部分，被拼接到 System Prompt 里发给模型。模型用和读用户消息一样的方式"读"工具定义。所以 description 的好坏直接决定了模型能不能正确使用这个工具。

二、技能：好的 description 怎么写？

2.1 三要素公式

每个 description 应该包含三个要素：

📝 Description 三要素
  1. 做什么：这个工具的功能，一句话说清楚
  2. 返回什么：调用后会得到什么结果（格式、内容）
  3. 怎么用：参数格式、示例值、边界情况

2.2 对比：好描述 vs 坏描述

评级 内容 问题
❌ 差 "获取天气" 没说返回什么、参数格式。模型可能传 "Beijing" 而不是 "北京"
⚠️ 一般 "获取指定城市的天气" 知道要传城市，但不知道格式。没有返回值说明
✅ 好 "获取指定城市的当前天气信息，返回温度、湿度、天气状况。城市名用中文，如 '北京'、'上海'。" 功能明确 + 返回内容 + 参数格式 + 示例值

2.3 参数 description 同样重要

每个参数的 description 也要写好——模型是逐个参数"理解"的：

// ❌ 坏：参数描述太模糊
"city": {
  "type": "string",
  "description": "城市"        // ← 模型不知道要传中文还是英文
}

// ✅ 好：带格式说明和示例
"city": {
  "type": "string",
  "description": "城市名称，用中文。支持：北京、上海、深圳、杭州、成都"
}
⚠️ 常见坑：description 写了但参数没有

函数级的 description 写了"城市名用中文"，但参数级的 description 只写了"城市"——模型可能只看参数描述，照样传错。两个层级的 description 都要写好。

三、技能：TypeScript 类型 → JSON Schema 映射

作为前端开发者，你已经习惯了 TypeScript 的类型系统。JSON Schema 是另一套"类型语言"，需要做映射：

TypeScript JSON Schema 备注
string {"type": "string"} 可加 enum 限制可选值
number {"type": "number"} 也可用 "integer"
boolean {"type": "boolean"}
string[] {"type": "array", "items": {"type": "string"}} 数组用 items 定义元素类型
{ city: string } {"type": "object", "properties": {...}} 对象展开为 properties
city?: string 不在 required 数组中 可选 = 不放 required
type Color = 'red' \| 'blue' {"type": "string", "enum": ["red", "blue"]} 联合字符串字面量 → enum
💡 前端直觉：JSON Schema ≈ PropTypes

如果你用过 React 的 PropTypes，JSON Schema 就是同一件事——描述一个对象的"形状"。区别是 PropTypes 给 React 用，JSON Schema 给 LLM 看。

四、动手：写一个完整的工具定义

✏️ 练习：写一个 searchCode 工具

请为以下 TypeScript 函数写出 JSON Schema 工具定义：

// 在项目代码库中搜索
function searchCode(query: string, lang?: string, maxResults?: number): SearchResult[]
  • query：搜索关键词，必填
  • lang：编程语言过滤（如 'ts'、'js'），可选
  • maxResults：最多返回条数，可选，默认 10
📝 参考答案 
{
  type: "function",
  function: {
    name: "searchCode",
    description:
      "在项目代码库中搜索指定关键词，返回匹配的文件路径、行号和代码片段。" +
      "适用于查找函数定义、API 调用、类型定义等。例如搜索 'useState' 找到所有使用该 Hook 的组件。",
    parameters: {
      type: "object",
      properties: {
        query: {
          type: "string",
          description: "搜索关键词，支持部分匹配。如 'useEffect'、'import.*from'",
        },
        lang: {
          type: "string",
          enum: ["ts", "tsx", "js", "jsx", "json", "css"],
          description: "限定编程语言，不传则搜索所有文件类型。可选值：ts, tsx, js, jsx, json, css",
        },
        maxResults: {
          type: "integer",
          description: "最多返回的搜索结果条数，默认 10。范围 1-50",
        },
      },
      required: ["query"],
    },
  },
}

检查清单：

  • ✅ 函数 description 包含三要素：做什么 + 返回什么 + 怎么用（带例子）
  • ✅ query 的 description 写了匹配方式
  • ✅ lang 用了 enum 限制可选值，减少模型传错的可能
  • ✅ maxResults 用 integer 而非 number，写了默认值和范围
  • ✅ 只有 query 在 required 中，另外两个是可选的

五、System Prompt 配合工具定义

除了工具本身的 description，你还可以通过 System Prompt 引导模型的行为：

const SYSTEM_PROMPT = `你是一个智能助手，可以调用工具来获取实时信息。

工具使用规则：
1. 当需要实时数据时，必须调用工具，不要编造
2. 当缺少工具的必填参数时，先向用户询问，不要猜测
3. 拿到工具结果后，用自然语言简洁地回复用户
4. 如果工具返回错误，如实告知用户，不要掩饰`;
⚠️ 关键规则：缺少参数时先问，不要猜

如果用户问"天气怎么样"而没指定城市，模型可能会"猜"一个城市（通常是北京）。这会导致不准确的结果。在 System Prompt 中明确要求"缺少必填参数时先询问用户"，可以大幅减少这种问题。

六、自测

5 题，覆盖工具定义的核心技能。

  1. 为什么工具定义的 description 要写"返回什么"而不仅仅是"做什么"？
  2. 一个工具有 5 个参数，但只有 2 个是必填的。在 JSON Schema 中如何表达？
  3. 参数类型用 "integer" 和 "number" 有什么区别？什么时候该用哪个？
  4. 为什么要在 System Prompt 中写工具使用规则，而不是只靠工具定义？
  5. 如果模型反复调用同一个工具但传错参数（比如城市名用英文），你应该检查什么？
📝 参考答案

第 1 题：为什么 description 要写"返回什么"？

因为模型需要知道工具返回的数据长什么样，才能正确地将结果"翻译"给用户。

模型在收到工具结果后，需要决定如何向用户呈现。如果它不知道返回内容的格式，可能会出现：

  • 忽略关键信息（比如温度数据里包含了降雨概率，但模型没提）
  • 错误解读数据（把湿度当成温度）
  • 过度简化（返回了 5 个字段，模型只用了 1 个）

底层机制：模型在 Prefill 阶段"阅读"了整个 Prompt（包括工具定义），然后基于所有信息预测下一个 token。如果工具定义里写了"返回温度、湿度、天气状况"，这些词就会出现在模型的注意力窗口里，模型在生成回答时更容易"注意到"这些信息并正确使用。

类比：就像调用 API 时看文档——你不仅要看"这个 API 干嘛的"，还要看"返回的 JSON 结构是什么"，才能在前端正确渲染。模型也一样。

第 2 题：如何表达"5 个参数但只有 2 个必填"？

只把必填参数名放进 required 数组。不在 required 里的参数，模型会理解为"可选"。

// 5 个参数，只有 query 和 userId 是必填
parameters: {
  type: "object",
  properties: {
    query:    { type: "string", description: "搜索关键词" },       // 必填
    userId:   { type: "string", description: "用户 ID" },           // 必填
    limit:    { type: "integer", description: "返回条数，默认 20" },  // 可选
    offset:   { type: "integer", description: "分页偏移，默认 0" },   // 可选
    sortBy:   { type: "string", description: "排序字段" },           // 可选
  },
  required: ["query", "userId"],  // ← 只有这两个必填
}

行为差异：

  • 必填参数缺失 → 模型应该询问用户（如果 System Prompt 写得好）或猜测一个值（如果没写）
  • 可选参数缺失 → 模型直接跳过，不传这个参数

设计原则：只把"没有它工具就无法执行"的参数设为必填。如果一个参数有合理的默认值，让它可选，在 description 中说明默认值。

第 3 题：integer vs number 的区别和使用场景

核心区别：integer 只接受整数，number 接受整数和浮点数。这直接影响模型传参的行为。

场景 应该用 原因
返回条数 (limit) integer 不可能是 10.5 条
温度值 number 可能是 25.5°C
页码 (page) integer 页码总是整数
金额 number 有小数（元.角.分）

判断标准：问自己"这个参数的值可能包含小数点吗？"——可能 → number；不可能 → integer。用对类型可以减少模型传错参数的概率。

第 4 题：为什么需要 System Prompt + 工具定义？

因为工具定义告诉模型"能做什么"，System Prompt 告诉模型"该怎么做"。

工具定义是"说明书"——描述工具的接口（输入输出）。System Prompt 是"工作手册"——描述使用工具的规则和策略。两者解决不同层面的问题：

解决的问题 在哪里定义
这个工具叫什么、需要什么参数 工具定义 (tools)
什么时候该用工具、什么时候不该用 System Prompt
缺少参数时应该询问还是猜测 System Prompt
工具返回错误时怎么处理 System Prompt
回答问题应该用什么风格（简洁/详细） System Prompt

具体例子：工具定义里写了 city 是必填参数。但用户问"天气怎么样"时没指定城市。模型有两种选择：① 猜一个城市（可能猜错）② 反问用户"你想查哪个城市？"。System Prompt 中的"缺少必填参数时先询问用户"就告诉模型选②而不是①。

类比：工具定义 = API 文档（接口签名），System Prompt = 团队规范（什么时候调用、怎么处理异常）。两者缺一不可。

第 5 题：模型反复传错参数，应该检查什么？

按优先级依次检查以下 4 个地方：

  1. 参数 description 是否写了格式要求？

    如果模型传 "Beijing" 而不是 "北京"，检查参数 description 是否明确写了"城市名用中文"。参数级的 description 比函数级的更直接影响模型行为——模型在"填参数"这一步主要看参数描述。

  2. 函数 description 是否有冲突信息？

    如果函数描述说"支持全球城市"，参数描述说"仅中国城市"，模型会困惑。两个层级的描述要一致。

  3. 是否用了 enum 限制可选值？

    如果参数只有有限的合法值（如城市名），用 enum 而不是只靠 description。模型看到 enum 会更严格地选择。

  4. System Prompt 是否缺少行为引导？

    如果以上都检查了还是不行，在 System Prompt 中添加更具体的行为规则，如"getWeather 的 city 参数必须是中文城市名，如 '北京'，不要用拼音或英文"。

调试技巧：打印模型返回的原始 tool_call JSON，看看它到底传了什么参数。然后针对性地修改 description。不要猜测——看数据。
