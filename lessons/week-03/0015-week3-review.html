🔑 关键设计决策
  • messages 数组贯穿始终：system → user → assistant(tool_call) → tool(result) → assistant(tool_call) → ... → assistant(answer)。每一步都 push 进数组，模型每次都"读"完整历史。
  • AgentState 追踪一切：step 计数、toolCallsMade 记录（工具名 + 参数 + 结果）、finalAnswer、总耗时。状态透明，出问题能回溯。
  • 并行检测在代码层：如果 toolCalls.length > 1 → Promise.all 同时执行。模型不需要知道"并行"是什么——它只管一次返回多个 tool_call。
  • System Prompt 是工具选择的"说明书"：5 个工具靠 description 区分还不够，System Prompt 给出了明确的规则——"项目代码 → searchCode，外部知识 → searchWeb"。
  • MAX_STEPS = 6：防止无限循环。达到上限后强制模型基于已有信息总结，不再调工具。

四、本周常见踩坑总结

⚠️ 第 3 周踩坑记录
  1. 小模型（3b）并行能力弱：qwen2.5:3b 面对"同时查北京和上海天气"经常只返回 1 个 tool_call。不是代码问题，是模型能力限制。8b 模型通常能正确并行。
  2. 工具描述太像导致选错：searchWeb 和 searchCode 都叫"搜索"，模型容易混淆。解决：在 description 中明确写"适用于 X / 不适用于 Y"，并在 System Prompt 中补充规则。
  3. System Prompt 缺失导致工具不触发：场景 1/2/3 最初只传了用户消息，没带 System Prompt。模型看到"项目里 XX 接口怎么定义"可能当成通用知识直接回答。加上 System Prompt 后准确率从 4/5 提升到 5/5。
  4. 工具执行失败可能导致死循环：模型调了一个不存在的工具 → 返回错误 → 模型又调 → 又错误。必须设 MAX_STEPS 作为安全护栏。
  5. ESM 下 __dirname 不可用：tsx 以 ESM 模式运行，__dirname 是 undefined。需要用 import.meta.url + fileURLToPath + path.dirname 替代。
  6. 模型可能"幻觉调用"不存在的工具：如果 tool 定义不够清晰，模型可能编造一个不存在的工具名。executeTool 中必须有 default 分支处理未知工具。

📝 参考答案

第 1 题：Function Calling 的本质

模型没有调用任何函数。它只是输出了一段结构化的 JSON 数据（tool_call），包含三个字段：

  • id：本次调用的唯一标识（用于后续关联 tool result）
  • function.name：想调用的函数名
  • function.arguments：传给函数的参数（JSON 对象）

真正执行函数的是你的代码。流程是：

  1. 代码收到模型的 tool_call JSON
  2. 代码解析 JSON，提取函数名和参数
  3. 代码调用真正的 TypeScript 函数（如 getWeather("北京")）
  4. 代码把函数返回值包装成 tool role 消息，回传给模型

为什么这种设计是合理的？

  • 安全：模型不能执行任意代码——它只能"建议"调哪个函数，代码决定是否执行。如果模型可以真正调用函数，它就能删文件、发网络请求、执行恶意代码。
  • 解耦：模型只负责推理和决策（"这个城市需要查天气"），不关心函数怎么实现（API 调用还是数据库查询）。你可以在不改模型的前提下切换工具的实现。
  • 可控：代码可以在执行前做校验（参数是否合法？频率是否超限？），模型不需要知道这些限制。

前端类比：就像 React 组件不直接操作 DOM——它输出 Virtual DOM（声明式描述），React 运行时负责把 Virtual DOM 变成真实 DOM。模型输出 tool_call（声明式描述），你的代码负责执行。

第 2 题：description 为什么比 parameters 更重要？

核心原因：模型靠 description 决定"要不要调这个工具"，parameters 只影响"调的时候传什么参数"。如果 description 写得不好，模型根本不会选择这个工具，parameters 写得再好也没用。

对比例子：

❌ 差的 description
description: "搜索信息"

问题：太模糊，模型不知道和 searchCode 有什么区别。两个工具都"搜索信息"，选哪个？

✅ 好的 description
description: "搜索互联网获取最新信息。适用于：模型知识截止后的内容、最新技术文档、实时资讯。不适用于：查找项目代码、本地文件。与 searchCode 的区别：searchWeb 查外部知识，searchCode 查项目本地代码。"

明确了三要素：① 做什么（搜索互联网）② 何时用（知识截止后、最新资讯）③ 何时不用（项目代码、本地文件）。模型能准确判断。

Description 三要素（必须包含）：

  1. 做什么：工具的功能描述
  2. 何时用：适用场景（"适用于 X"）
  3. 何时不用：不适用场景 + 与其他工具的区别（"不适用于 Y，与 Z 的区别是..."）

第 3 题：MAX_STEPS 的必要性

为什么需要 MAX_STEPS？

Agent 循环是一个 while 循环——模型每次决策是否调工具。如果模型陷入困惑（工具返回错误、结果不理想、模型幻觉），它可能不断尝试调工具，形成无限循环。MAX_STEPS 是硬性终止条件。

太小的后果：

  • 复杂任务（需要多步推理）被强制中断，用户得到不完整的答案
  • 例如"查 A 天气，如果下雨就查 B 天气，如果也下雨就算 C"——这个任务需要 3-4 步，MAX_STEPS=2 会提前终止

太大的后果：

  • 模型陷入死循环时浪费大量 token 和时间
  • 用户体验差——等了很久才发现 Agent 在兜圈子
  • 一般设 5-8 步，Demo 2 中设的是 6

最佳实践：达到 MAX_STEPS 后，不要直接返回"失败"，而是给模型一个最终提示："请基于已有信息给出最终回答，不要再调用工具。"——让模型总结已有信息，用户至少能得到一个部分答案。

第 4 题：并行 vs 串行的判断标准

判断标准：问自己一个问题——"任务 B 的输入是否依赖任务 A 的输出？"

  • 依赖 → 串行：必须先完成 A，拿到结果，才能决定 B 要不要做、怎么做。
  • 不依赖 → 并行：A 和 B 可以同时做，互不影响。
场景 策略 原因
"查北京和上海的天气" ⚡ 并行 两个查询互不依赖
"如果深圳下雨就提醒我带伞" 🔗 串行 "是否提醒"依赖"是否下雨"的结果
"现在几点了？顺便算 1024×768" ⚡ 并行 两个任务完全独立
"搜 React 19 新特性，然后总结" 🔗 串行 "总结"依赖"搜索结果"

代码层面的区别：

  • 并行：Promise.all(toolCalls.map(tc => executeTool(...)))——所有工具同时执行，结果一起回传
  • 串行：executeTool(toolCalls[0]) → 回传 → 下一轮 ollama.chat → 可能再调下一个工具

注意：并行调用是小模型的弱项。qwen2.5:3b 面对"同时查北京和上海"经常只返回 1 个 tool_call。这不是代码 bug，是模型能力限制。

第 5 题：messages 数组中的 role 类型

Demo 2 中 messages 包含 4 种 role，顺序极其重要：

role 作用 示例
system 设定角色和工具选择规则 "你是一个智能助手，可以调用工具..."
user 用户的问题或指令 "北京今天天气怎么样？"
assistant 模型的回复（文本或 tool_call） { content: "", tool_calls: [...] }
tool 工具执行的结果，必须关联 tool_call_id "晴天，25°C，湿度 45%"，tool_call_id: "call_abc"

为什么顺序重要？模型是按顺序"阅读" messages 的。tool 消息必须紧跟在对应的 assistant(tool_call) 消息之后，模型才能把"我让你做的事"和"你做完了的结果"关联起来。如果 tool 消息的 tool_call_id 对不上，模型会困惑。

典型的消息序列：

system    →  设定规则
user      →  "查北京天气"
assistant →  tool_calls: [getWeather("北京")]
tool      →  "晴天 25°C" (tool_call_id: call_abc)
assistant →  "北京今天晴天，25°C..."（最终回答）
