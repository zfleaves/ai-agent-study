一、Day 1 的遗留问题：messages 数组无限增长

回顾 Day 1 的 ReAct 循环代码：

const messages = [
  { role: 'system', content: SYSTEM_PROMPT },
  { role: 'user', content: prompt },
];

while (step < MAX_STEPS) {
  const res = await client.chat.completions.create({ messages, tools });
  const msg = res.choices[0]?.message;

  messages.push(msg);           // ← 助手消息
  for (const tc of msg.tool_calls) {
    const result = executeTool(tc);
    messages.push({ role: 'tool', content: result, tool_call_id: tc.id });  // ← 工具结果
  }
}
⚠️ 三个隐患
  1. 消息无限增长：每轮对话 push 2-3 条消息，100 轮后就是 300 条。LLM 有上下文窗口限制（如 8K/32K/128K tokens），超出后 API 直接报错。
  2. 成本线性增长：每次请求都发送完整 messages 数组，token 消耗随对话轮次线性增长，越聊越贵。
  3. 无持久化：进程退出后消息丢失，用户刷新页面就丢失全部对话。

二、消息角色：Agent 对话的"语法"

2.1 四种角色及其职责

LLM API 的消息不是简单的"用户说了什么、AI 回了什么"，而是有四种严格定义的角色。理解它们是用好 Agent 的前提。

角色 英文 发送者 职责 何时出现
system 系统 开发者 设定 Agent 的行为规则、角色定位、输出格式 对话开始时（通常只有 1 条）
user 用户 用户 用户的问题、指令、输入 每轮对话开始时
assistant 助手 LLM 模型的回复，可能包含 content（文本）和 tool_calls（工具调用） 每轮 LLM 响应后
tool 工具 你的代码 工具执行的结果，必须通过 tool_call_id 关联到对应的 assistant tool_calls 每次工具执行后

2.2 角色间的依赖关系

┌─────────────────────────────────────────────────────────────┐
│                    消息角色关系图                              │
│                                                             │
│  system (规则) ───────────────────────────────────────┐     │
│    │                                                  │     │
│    ▼                                                  │     │
│  user (问题)                                          │     │
│    │                                                  │     │
│    ▼                                                  │     │
│  assistant (回答 + tool_calls) ◄── 始终参考 system ───┘     │
│    │                                                       │
│    ├── content: "我来查一下..."                              │
│    └── tool_calls: [{id: "call_1", ...}]                    │
│         │                                                   │
│         ▼                                                   │
│  tool (结果) ─── tool_call_id = "call_1" ─── 必须配对！      │
│    │                                                       │
│    ▼                                                       │
│  assistant (最终回答)                                        │
│                                                             │
│  ⚠️ 关键约束：                                               │
│  • tool 消息必须紧跟 assistant 的 tool_calls，不能独立出现     │
│  • 截断时不能拆散 assistant(tool_calls) + tool 这对           │
│  • system 消息通常应该始终保留，不被截断                       │
└─────────────────────────────────────────────────────────────┘
⚠️ 常见错误：tool 消息孤岛

如果截断时删除了 assistant（含 tool_calls）但保留了 tool 消息，API 会报错：

// ❌ 错误：tool 消息孤立，没有对应的 assistant tool_calls
messages = [
  { role: 'tool', content: '晴天，25°C', tool_call_id: 'call_1' },
  // 找不到 role=assistant 且 tool_calls[].id === 'call_1' 的消息
]

这就是为什么截断策略必须"成对处理" —— assistant(tool_calls) 和 tool 消息要么一起保留，要么一起删除。

三、State 设计：消息不只是 Array

3.1 从"裸数组"到"结构化状态"

维度 裸数组 (Day 1) AgentState Class (Day 2)
消息存储 any[] — 无类型安全 AgentMessage[] — 严格类型
系统消息 混在数组里，容易丢失 独立字段，始终保留
步数追踪 let step = 0 — 手动维护 incrementStep() — 自动管理
Token 估算 无 estimateTokens()
截断策略 无 3 种策略，可配置
持久化 无 toJSON() / fromJSON() / saveToFile()
查询 messages.filter(...) — 每次手写 getRecent(n) / getByRole(role)

3.2 AgentState 的核心方法

class AgentState {
  // ── 消息管理 ──
  setSystem(content: string)           // 设置系统消息（独立存储）
  addUserMessage(content: string)      // 添加用户消息
  addAssistantMessage(content, tc?)    // 添加助手消息（含 tool_calls）
  addToolResult(toolCallId, content)   // 添加工具返回消息
  incrementStep()                      // 步数 +1

  // ── 查询方法 ──
  getMessages()                        // 获取完整消息列表（发给 LLM）
  getRecent(n: number)                 // 获取最近 N 条
  getByRole(role: MessageRole)         // 按角色筛选
  estimateTokens(text: string)         // 估算 token 数
  estimateTotalTokens()                // 估算总 token 数

  // ── 截断策略 ──
  setTruncation(config)                // 配置截断策略
  truncate()                           // 执行截断

  // ── 持久化 ──
  toJSON() / fromJSON(json)            // 序列化 / 反序列化
  saveToFile(path) / loadFromFile(path) // 文件读写
}

四、历史截断：三种策略对比

4.1 为什么需要截断？

LLM 的上下文窗口（Context Window）是有限的。即使是最新的模型（128K tokens），长对话也会逼近上限。更重要的是：每次请求都发送完整历史，token 消耗 = 对话轮次 × 每轮 token 数。100 轮对话可能消耗数十万 tokens。

💡 前端直觉类比

消息历史就像 React 组件的 state——如果每次 render 都累积旧 state 而不清理，内存会无限增长。截断策略就像 useMemo 和 useCallback：只保留"当前渲染需要"的数据，旧数据要么丢弃，要么压缩缓存。

4.2 三种策略对比

策略 原理 ✅ 适用场景 ❌ 不适用场景 计算成本
Sliding Window
滑动窗口		 只保留最近 N 条消息，旧消息直接丢弃 短期任务、单轮对话、不需要上下文的任务 多步推理需参考早期信息、长文档分析 极低（数组 slice）
Keep System + Last
保系统+最近		 始终保留 system 消息，从头部删除旧消息直到 token 数低于阈值。成对处理 assistant/tool 最常用——保留对话规则 + 最近上下文，适合大多数 Agent 任务 需要长期记忆的对话（如 3 小时前的用户偏好） 低（token 估算 + 循环删除）
Summarize
摘要压缩		 将旧消息压缩为一条摘要，保留最近 N 条完整消息 长期对话、需要"记忆"早期关键信息、客服/陪伴类 Agent 摘要可能丢失细节、需要精确引用早期对话的场景 高（需调用 LLM 生成摘要）

4.3 截断的核心约束：tool_calls 成对原则

// ❌ 错误截断：拆散了 assistant(tool_calls) 和 tool 消息
messages = [
  { role: 'assistant', tool_calls: [{id: 'call_1', ...}] },  // ← 被保留
  { role: 'tool', tool_call_id: 'call_1', content: '...' },  // ← 被删除！
]
// API 报错：tool 消息对应的 tool_call_id 找不到

// ✅ 正确截断：成对处理
messages = [
  // assistant(tool_calls) + tool 要么一起保留，要么一起删除
]
✅ 截断安全检查清单
  1. system 消息是否始终保留？
  2. 是否存在孤立的 tool 消息（无对应 assistant tool_calls）？
  3. 是否存在孤立的 assistant tool_calls（无对应 tool 结果）？
  4. 截断后总 token 是否低于目标阈值？
  5. 截断后是否有足够的上下文让模型理解当前对话？

六、Token 估算：为什么不用精确计数？

6.1 精确计数的代价

精确计算 token 数需要调用 tokenizer（如 tiktoken 库），但有成本：

方式 准确性 依赖 性能 推荐场景
字符估算 ±20% 无 极快 截断判断（不需要精确）
tiktoken 精确 tiktoken 包 + 模型 tokenizer 快 成本核算、计费
API 返回 精确 API 响应 事后 实际消耗统计

对于截断判断，±20% 的误差完全可接受。我们只需要在逼近上限前触发截断，不需要精确到个位数。

6.2 估算规则

estimateTokens(text: string): number {
  // 中文：~1.5 字符/token（中文 tokenizer 效率高）
  // 英文/代码：~4 字符/token
  let chineseChars = 0, otherChars = 0;
  for (const ch of text) {
    if (/[一-鿿]/.test(ch)) chineseChars++;
    else otherChars++;
  }
  return Math.ceil(chineseChars / 1.5 + otherChars / 4);
}

七、序列化：让会话"活过"进程重启

7.1 为什么需要序列化？

Web 应用中，用户刷新页面 → 进程重启 → 所有内存中的对话历史丢失。如果用户正在和 Agent 进行一个 30 轮的复杂任务，这会非常糟糕。

解决方案：将 AgentState 序列化为 JSON，持久化到文件或数据库。

// 保存
state.saveToFile('./sessions/latest.json');

// 恢复
const state = AgentState.loadFromFile('./sessions/latest.json');

// 数据库存储（伪代码）
await db.sessions.upsert({
  id: state.sessionId,
  data: state.toJSON(),
  updatedAt: new Date(),
});

7.2 序列化要考虑的边界情况

  • 向后兼容：如果 AgentState 加了新字段，旧 JSON 能正常加载吗？→ 用版本号标记
  • 安全性：JSON 中可能包含敏感信息（用户输入、工具调用参数）→ 加密或脱敏
  • 大小：长对话的 JSON 可能很大（数百 KB）→ 结合截断策略，定期清理

📖 答案详解

第 1 题：消息角色

正确答案：B

合法序列必须是：system（可选）→ user → assistant（可能含 tool_calls）→ tool（如有 tool_calls）→ assistant → user → ... 循环。

  • A 为什么错：tool 消息不能出现在 assistant 之前。tool 消息必须有 tool_call_id，这个 ID 来自 assistant 的 tool_calls。如果把 tool 放在 assistant 前面，API 找不到对应的 tool_call_id，直接报错。
  • B 为什么对：完美符合规范。system 设定规则 → user 提问 → assistant 决定调用工具（含 tool_calls）→ tool 返回结果 → assistant 基于结果给出最终回答。这是最标准的 Agent 一轮对话流程。
  • C 为什么错：assistant(tool_calls) 之后必须是 tool 消息，不能直接跳到 user。因为 assistant 说"我要调用工具 X"，你必须给工具执行结果（tool 消息），然后模型才能继续。跳过这步等于"承诺了但没兑现"。
  • D 为什么错：assistant 不能出现在 user 之前。对话必须由用户发起（或 system 之后第一个是 user）。这就像你不能在别人提问之前就回答。

第 2 题：截断策略选择

正确答案：C

客服场景的关键需求是"长期记忆"——用户可能在对话中随时提到早期信息（如订单号），Agent 需要能"记住"。

  • A 为什么不对：Sliding Window 只保留最近 10 条消息，30 分钟前的订单号信息早就被丢弃了。当用户说"我之前提到的那个订单"，Agent 会一脸茫然。
  • B 为什么不对：Keep System + Last 同样只保留最近消息，早期信息会被 token 限制触发截断而删除。它适合"当前任务上下文"而非"长期记忆"。
  • C 为什么对：Summarize 将早期对话压缩为摘要。虽然摘要会丢失细节，但"订单号 2024-X"这类关键信息能在摘要中保留。当用户再次提及时，Agent 能从摘要中找到线索。这也是 RAG 的简化版——用摘要代替向量检索。
  • D 为什么不对：不截断在短对话中可行，但客服对话可能持续数小时、数百轮。最终会超出上下文窗口，API 直接报错。而且不截断意味着每次请求都发送完整历史，token 成本线性增长。

第 3 题：tool_calls 成对原则

正确答案：B

这是 API 的硬性约束。每个 tool 消息的 tool_call_id 字段必须能匹配到前面某条 assistant 消息的 tool_calls[].id。

  • 底层原因：LLM 在处理消息时，需要知道每条 tool 消息是"哪个工具调用的结果"。如果 tool_call_id 找不到对应的 assistant tool_calls，模型不知道这个结果从哪来的，无法正确推理。这就像你收到了一个快递但不知道是哪个订单的——你没法确认收货。
  • 为什么不是 A：虽然成对删除确实效率更高，但这不是主要原因。即使不成对删除效率低，只要 API 不报错，也可以接受。问题是 API 直接报错，不是因为效率。
  • 为什么不是 C：美观与 API 规范无关，API 不关心消息好不好看。
  • 为什么不是 D：不成对的消息不会导致推理变慢——它会导致 API 直接返回 400 错误，根本不会开始推理。

第 4 题：system 消息的特殊性

正确答案：B

system 消息是 Agent 的"宪法"——它定义了 Agent 是谁、能做什么、不能做什么、如何输出。无论对话历史如何截断，这些规则都不能丢失。

  • 底层原因：如果 system 消息被截断删除了，Agent 会失去行为约束。比如 system 说"你只能回答技术问题，拒绝政治话题"，如果这条被删了，Agent 可能开始回答不该回答的问题。这不仅是功能问题，也是安全问题。
  • 为什么不是 A：system 消息通常并不比用户消息更长（一般 200-500 字），空间不是主要考虑。
  • 为什么不是 C：system 消息就是普通文本，没有任何二进制格式。
  • 为什么不是 D：API 不要求 system 消息单独发送——它就在 messages 数组的第一条。独立存储是我们的设计选择，不是 API 的要求。

第 5 题：Token 估算

正确答案：B

中文的 tokenizer 效率确实更高。原因在于 tokenizer 的训练数据分布和分词粒度。

  • 底层原理：BPE（Byte Pair Encoding）tokenizer 在训练时，高频字符组合会合并为一个 token。中文每个字通常是一个独立的语义单元，且常用字频率高，所以往往 1-2 个中文字符 = 1 个 token。英文单词由多个字母组成，平均 4 个字符 = 1 个 token。这就是为什么同样长度的文本，中文的 token 数通常更少。
  • 为什么不是 A：字符估算虽然不精确（±20%），但对于截断判断完全够用。我们只需要在逼近上限前触发截断，不需要精确到个位数。就像你不需要精确知道油箱还剩多少升——知道大概还剩 1/4 就够你决定去加油了。
  • 为什么不是 C：不同语言的 tokenizer 效率差异很大。中文 ~1.5 字符/token，英文 ~4 字符/token，日文介于两者之间。统一用 4 字符/token 会严重低估中文文本的 token 数。
  • 为什么不是 D：等 API 报错再截断的问题是：① 用户体验差（突然报错 vs 平滑截断）；② 可能丢失最关键的消息（报错时已经来不及智能选择了）；③ 浪费了这次请求的 token 成本。
