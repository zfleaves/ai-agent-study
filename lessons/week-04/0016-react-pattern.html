一、第 3 周 Agent 的"天花板"

第 3 周我们做出了一个能调工具的 Agent。用户可以问：

用户："查北京天气" → Agent 调 getWeather → 返回结果 ✅

但面对这种问题呢？

用户："帮我查一下北京天气，如果下雨就提醒我带伞，然后查明天的高铁到上海"

Agent 需要：
  1. 先查北京天气
  2. 读结果，判断是否下雨
  3. 如果下雨 → 提醒带伞
  4. 然后查高铁
  5. 整合所有信息，给出最终回答
⚠️ 第 3 周 Agent 的局限性

第 3 周的 Agent 循环核心逻辑是：

while (step < MAX_STEPS) {
  模型响应 → 有 tool_call？ → 执行 → 回传 → 继续
}

这个循环能"调工具"，但缺少一个关键环节：模型在调工具之前，没有显式地思考"为什么调这个工具"和"调完要达到什么目的"。当任务涉及多步推理和条件判断时，模型容易迷失方向——它只是在机械地"调工具→看结果→调下一个"，而不是在"想清楚再行动"。

二、ReAct 是什么？

2.1 核心思想：Thought → Action → Observation

ReAct = Reasoning + Acting，是 Yao et al. 在 2022 年提出的 Agent 推理框架。核心思想是：Agent 在每次行动前，先输出一个"思考"（Thought），解释为什么要做这个行动、期望达到什么效果。

┌─────────────────────────────────────────────────────────┐
│                    ReAct 循环                             │
│                                                         │
│   Thought → Action → Observation → Thought → Action → │
│     思考      行动       观察        再思考    再行动     │
│                                                         │
│   例：                                                  │
│   Thought: "用户想知道北京天气，我需要查天气数据"         │
│   Action:  getWeather("北京")                           │
│   Observation: "晴天，25°C"                             │
│   Thought: "天气好，不需要提醒带伞。接下来查高铁"         │
│   Action:  searchWeb("北京到上海高铁")                   │
│   Observation: "G123 次，09:00-13:30"                   │
│   Thought: "信息齐全了，可以回答用户"                    │
│   Answer:  "北京今天晴天 25°C，不需要带伞..."            │
└─────────────────────────────────────────────────────────┘

2.2 对比：有 Thought vs 没有 Thought

维度 纯 Tool-calling（第 3 周） ReAct（本课）
流程 用户提问 → 模型调工具 → 返回结果 用户提问 → 模型思考 → 调工具 → 观察结果 → 再思考 → ...
模型输出 只输出 tool_call 或 final answer 输出 Thought + tool_call 或 final answer
多步推理 弱——模型可能"调了再说" 强——每步都有明确的推理目标
条件判断 隐式——模型心里知道，但没写出来 显式——"如果A就B"写在 Thought 里
可调试性 差——不知道模型为什么选错工具 好——Thought 告诉你模型的推理过程
Token 消耗 少 多（Thought 也消耗 token）

2.3 前端直觉类比

// 纯 Tool-calling ≈ 直接调 API，不写注释
async function handleUserRequest(query: string) {
  const tool = pickTool(query);        // 黑盒决策
  const result = await execute(tool);  // 执行
  return format(result);               // 返回
}

// ReAct ≈ 每步都写 console.log + 注释，解释"为什么"
async function handleUserRequestReAct(query: string) {
  console.log("Thought: 需要先查天气，判断是否下雨");
  //  ↑ 显式思考，出问题时能看到推理过程
  const weather = await getWeather("北京");

  console.log("Observation: 天气是晴天，不需要提醒带伞");
  //  ↑ 观察结果，决定下一步

  console.log("Thought: 天气没问题，接下来查高铁");
  const train = await searchWeb("北京到上海高铁");

  console.log("Thought: 信息齐全，可以回答");
  return format(weather, train);
}

核心区别：ReAct 的 Thought 就像代码里的注释和日志——没有它代码也能跑，但有了它，你（和模型自己）就能看清楚每一步"为什么这么做"。

三、为什么 ReAct 有效？底层机制

3.1 自回归约束机制 — 为什么"写出来"比"心里想"好

这和第 2 周讲的 CoT（Chain of Thought，思维链）原理相同：

🔬 自回归约束（Autoregressive Constraint）

Transformer 模型是自回归的——生成每个 token 时，只能看到前面已经生成的 token。这意味着：

  1. 已生成的 token 会锁定后续 token 的概率分布。当模型写下"Thought: 用户想知道北京天气，如果下雨就需要提醒"时，这段话中的"下雨"、"提醒"等词会大幅提高后续生成"带伞"、"雨具"等词的概率。
  2. 没写出来的推理 ≈ 不存在。如果模型不在 Thought 中显式写下"如果下雨就..."，那这个条件判断对后续 token 的约束力就很弱——模型可能"忘了"要判断天气状况。
  3. Thought 是模型的"工作记忆"。人类做复杂推理时会自言自语或打草稿，模型也一样——把中间推理写出来，就相当于在 Context Window 中创建了一个"便签"，后续生成时随时可以参考。

3.2 ReAct 如何提升工具选择准确率

第 3 周我们遇到过一个场景：模型不调 searchCode 而是直接回答"接口是怎么定义的"。这是因为模型没有显式思考"这个问题需要查代码"。

在 ReAct 模式下：

❌ 纯 Tool-calling：
  用户："项目里 ToolDef 接口怎么定义的？"
  模型：（心里想）"我知道 interface 是什么，直接回答吧"
  输出：interface ToolDef { type: 'function'; ... }  ← 这是模型编的，不是项目实际代码

✅ ReAct：
  用户："项目里 ToolDef 接口怎么定义的？"
  模型：Thought: 用户在问项目代码中的接口定义，这需要查项目本地文件，不是外部知识
  Action: searchCode("ToolDef interface")
  Observation: agent-demo.ts:53  interface ToolDef { type: 'function'; ... }
  Thought: 找到了，在 agent-demo.ts 第 53 行
  Answer: 项目中 ToolDef 接口定义在 agent-demo.ts 第 53 行...

关键洞察：Thought 强制模型在行动前"解释自己的选择"，这个解释过程本身就纠正了很多选择错误。就像 code review 时口头解释自己的代码，往往说着说着就发现 bug 了。

四、ReAct 的 Prompt 设计

4.1 ReAct 格式规范

ReAct 论文定义了标准的输出格式。在 System Prompt 中告诉模型：

你是一个使用 ReAct 模式工作的智能助手。每次行动前，你必须先输出 Thought。

可用格式：
  Thought: <你的推理过程，解释为什么做这个行动>
  Action: <工具名>[<参数>]
  Observation: <工具返回的结果>

当信息足够时，输出：
  Thought: 信息齐全，可以回答
  Answer: <最终回答>

重要规则：
  1. Thought 必须在 Action 之前
  2. 每次 Action 后，等待 Observation 再继续
  3. 不要在 Thought 中猜测结果——等 Observation 后再判断

4.2 第 3 周 Agent vs ReAct Agent 的 System Prompt 对比

维度 第 3 周 System Prompt ReAct System Prompt
工具选择 "天气查询 → getWeather" "先 Think 再选工具，Think 里解释为什么选"
输出格式 tool_call 或纯文本 Thought + tool_call / Thought + Answer
终止条件 "直接回答" "Thought: 信息齐全 → Answer"
错误处理 "缺少参数时询问" "Thought: 工具返回错误，原因可能是... → 尝试替代方案"

5.2 观察：模型什么时候会"思考"？

跑几个测试，观察模型的行为：

问题类型 模型可能的行为 为什么
"北京天气" 无 Thought，直接调工具 任务简单，不需要显式推理
"如果深圳下雨就提醒我带伞" 可能有 Thought 有条件判断，需要先想清楚依赖关系
"查天气+搜新闻+计算" 可能有 Thought（并行分析） 需要判断哪些任务独立、哪些依赖

关键观察：小模型（3b）通常不会自动输出 Thought——它们倾向于直接行动。这正是 ReAct 格式的价值：用 System Prompt 强制模型输出 Thought，即使它"觉得"不需要。

六、ReAct 的适用场景与局限

场景 ReAct 合适吗？ 原因
多步推理（"先查A，如果A就B，再C"） ✅ 非常适合 Thought 帮助模型跟踪子目标
单工具调用（"查天气"） ⚠️ 过度设计 Thought 浪费 token，直接调工具即可
创意写作（"写一首诗"） ❌ 不适用 Thought 破坏流畅性，和 CoT 一样的问题
简单知识问答（"React 是什么"） ❌ 不适用 不需要工具，不需要推理
调试 Agent 行为 ✅ 非常有用 Thought 是天然的 debug 日志
⚠️ 判断标准：什么时候用 ReAct？

问自己两个问题：

  1. 这个任务需要 2 步以上的工具调用吗？ → 是 → 用 ReAct
  2. 步骤之间有条件依赖吗？（"如果A就B"）→ 是 → 必须用 ReAct

如果两个答案都是"否"，纯 Tool-calling 就够，不需要 ReAct 的 overhead。

七、ReAct vs 其他推理策略对比

策略 机制 工具调用 适用场景 Token 开销
Zero-shot 只给指令，不给例子 无需 简单任务、知识问答 最低
CoT 先推理再回答，思想链 无需 数学推理、逻辑分析 中等
Tool-calling 调工具，无显式推理 单步 查天气、搜索、计算 低
ReAct Thought → Action → Observation 多步 多步推理 + 工具调用 高
Plan-then-Act 先制定完整计划，再逐步执行 多步 复杂多步任务（Day 4 会讲） 最高

📝 参考答案

第 1 题：Thought、Action、Observation 的定义

三个要素：

  • Thought（思考）：模型在行动前输出的推理文本。解释"为什么做这个行动"、"期望达到什么效果"、"当前已知什么信息"。
  • Action（行动）：模型输出的工具调用（tool_call）。是具体的函数名 + 参数，由代码执行。
  • Observation（观察）：工具执行后返回的结果。是外部世界对 Action 的反馈，模型读取后决定下一步。

具体例子——"查北京天气，如果下雨就提醒带伞"：

① Thought:  "用户想知道北京天气，我需要先查天气数据，才能判断是否下雨"
② Action:   getWeather("北京")
③ Observation: "晴天，25°C，湿度 45%，风力 3 级"
④ Thought:  "天气是晴天，没有下雨。不需要提醒用户带伞。信息已经足够回答了。"
⑤ Answer:   "北京今天晴天，25°C，不需要带伞。"

顺序是关键：Thought 必须在 Action 之前，Observation 必须在 Action 之后。这个顺序保证了模型在行动前有明确的推理方向，在收到反馈后能重新评估。

第 2 题：ReAct 为什么比纯 Tool-calling 更适合多步推理？

核心原因：自回归约束机制。

Transformer 模型是自回归的——生成每个 token 时，只能看到前面已经生成的 token。这意味着：

  1. 已生成的 token 锁定后续概率分布：当模型写下"Thought: 需要先判断是否下雨，如果下雨就提醒用户带伞"时，这段文本中的"下雨"、"提醒"、"带伞"等词作为已生成的 token，会大幅提高后续生成相关内容的概率。模型在生成 Action 时，注意力机制会注意到这些词，从而更准确地选择正确的工具和参数。
  2. 纯 Tool-calling 缺少"中间记忆"：如果模型不写 Thought，它只能依赖前面几轮对话的上下文来做判断。对于多步推理（"如果A就B，否则C，然后D"），这个上下文链条太长，模型容易"忘记"前面的条件。
  3. Thought 是模型的"工作记忆"：每次 Thought 都相当于在 Context Window 中写下一个便签。后续步骤可以引用前面的 Thought，形成推理链条。就像人类做复杂数学题时需要在草稿纸上写中间步骤——不写就容易忘。

具体例子：

纯 Tool-calling（3 步任务）：
  Step 1: 调工具 → 结果
  Step 2: 调工具 → 结果  ← 模型可能已经忘了 Step 1 的目的是什么
  Step 3: 调工具 → 结果  ← 推理方向完全丢失

ReAct（3 步任务）：
  Step 1: Thought:"目标是X，先做A" → 调工具 → 结果
  Step 2: Thought:"A的结果是Y，还需要B" → 调工具 → 结果
  Step 3: Thought:"现在有了A和B的结果，可以回答X" → 回答
  ↑ 每一步都有明确的推理方向，Thought 之间形成逻辑链

第 3 题：ReAct 和 CoT 的区别

CoT（Chain of Thought） 和 ReAct 都利用了"思考 token 锁定后续分布"的自回归约束机制，但应用场景不同：

维度 CoT ReAct
核心能力 推理（Reasoning） 推理 + 行动（Reasoning + Acting）
是否需要工具 否 是
外部信息 只依赖模型内部知识 通过工具获取外部信息（Observation）
典型场景 数学题、逻辑推理、代码调试 多工具编排、条件判断、实时信息查询
输出格式 "让我们一步步思考..." → 最终答案 Thought → Action → Observation → ... → Answer

判断标准：问自己"这个推理过程需要获取外部信息吗？"

  • 不需要（如数学题：已知条件都在题目里）→ CoT 就够了
  • 需要（如查天气、搜索、读文件）→ 必须用 ReAct

ReAct 可以看作 CoT 的扩展——在 CoT 的推理链中插入了 Action 和 Observation 环节。CoT 是"纯思考"，ReAct 是"思考 + 行动 + 观察 + 再思考"。

第 4 题：什么时候不应该用 ReAct？

场景 1：单工具调用（"查北京天气"）

  • 原因：任务太简单，不需要多步推理。Thought 的 token 开销（50-100 tokens）相对于任务本身是不必要的。就像用大炮打蚊子——效率低下。
  • 代价：每次 Thought 消耗 50-100 tokens，如果一天 1000 次调用，多消耗 100k tokens。对于简单任务，这是纯浪费。
  • 替代方案：直接用纯 Tool-calling，或者只在首次调用失败时才启用 ReAct 重试。

场景 2：创意写作（"写一首关于春天的诗"）

  • 原因：和 CoT 一样，ReAct 的 Thought 会破坏输出的流畅性。创意写作需要的是"一气呵成"的连贯性，而不是"先分析再写"的机械感。
  • 代价：模型在 Thought 中分析"第一句应该用春天的意象，第二句应该转折..."，然后写出来的诗读起来像论文——失去了诗歌的韵律和情感。
  • 替代方案：Zero-shot 直接写，不满意再让模型自己修改。

通用判断标准：

if (任务需要工具调用 && 需要多步推理) {
  用 ReAct
} else if (任务需要工具调用 && 只需一步) {
  用纯 Tool-calling
} else if (任务不需要工具 && 需要推理) {
  用 CoT
} else {
  用 Zero-shot
}

第 5 题：强制 Thought vs 自主 Thought 的利弊

强制 Thought（System Prompt 要求"每次行动前必须输出 Thought"）：

利 弊
✅ 每次行动都有推理依据，可调试 ❌ 简单任务也强制输出 Thought，浪费 token
✅ 小模型也能正确推理（被强制思考） ❌ 增加 TTFT（Thought 也是生成的 token）
✅ 工具选择准确率更高 ❌ 可能导致"过度思考"——对简单问题分析太多

自主 Thought（让模型自己决定是否输出 Thought）：

利 弊
✅ 简单任务不浪费 token ❌ 小模型可能"偷懒"，不输出 Thought
✅ 响应更快（不需要生成 Thought token） ❌ 复杂任务可能选错工具（没有推理约束）
✅ 更自然——像人类一样，只在需要时思考 ❌ 不可调试——不知道模型为什么选错

最佳实践：开发阶段强制 Thought（方便调试），生产环境用自适应策略——简单任务跳过 Thought，复杂任务触发 ReAct。这需要 Agent 状态管理（Day 2 会讲）来实现。
