📖 第 2 课：LLM 核心概念

概念 第 1 周 Day 2 约 40 分钟

本课目标：搞懂 LLM 怎么"读"、怎么"想"、怎么"写"。建立 5 个核心概念：LLM 本质、Token、Context Window、Temperature、推理过程。

一、LLM 到底是什么？

一句话：LLM 是一个"猜下一个词"的概率机器。

它不是知识库，不是搜索引擎，它只是在做一件事：给定上文，预测下一个最可能出现的 token。

🔬 示例

你输入：  "中国的首都是"
模型计算：
  "北京" → 92%
  "上海" →  3%
  "南京" →  2%
  "深圳" →  1%
  ...
输出："北京"

然后拼回去："中国的首都是北京"
再猜下一个词 → <结束> → 停止

这个过程叫 自回归生成（Autoregressive Generation）——每一轮都把自己的输出拼回输入，继续预测下一个，直到输出结束符。

💡 关键洞察：LLM 没有"思考"，只有概率计算。但当参数足够大（billions of parameters），这种概率计算 涌现 出了看起来像推理的能力。

二、Token：模型"读"的最小单位

Token ≠ 字 ≠ 词。 Token 是模型内部表示文本的最小单位，由 tokenizer 切分产生。

🇨🇳 中文

"今天天气不错"
→ ["今天", "天气", "不错"]
→ 约 3 个 token

🇺🇸 英文

"The weather is nice today"
→ ["The", " weather", " is", " nice", " today"]
→ 5 个 token

经验法则：

  • 英文：1 token ≈ 0.75 个单词
  • 中文：1 token ≈ 0.5~0.7 个汉字
  • 代码：1 token ≈ 0.3~0.5 个字符（符号多，切得更碎）
💡 为什么重要：同样意思，中文消耗的 token 比英文多。API 按 token 计费，这就是为什么中文调用更贵。做 Agent 时，工具返回结果也要算 token 预算。

在线体验 Tokenizer

打开 OpenAI Tokenizer，输入一段中文和英文，数数各切了多少 token。

三、Context Window：模型的"视野"

模型一次能"看到"的内容是有上限的，这个上限就是 Context Window（上下文窗口）。

📐 Hermes 3 (8B) 的窗口

指标数值
Context Window约 8192 tokens
能装下约 20 页中文文档
能记住约 30-40 轮对话
装不下一本小说（《三体》≈ 40 万 tokens）

窗口溢出会怎样？

当输入超过 Context Window：

  1. API 直接报错 — 请求被拒绝
  2. 自动截断 — 模型只看到最后 N 个 token，前面的"遗忘"
  3. 性能下降 — 窗口边缘的信息容易被忽略（Lost in the Middle 现象）
💡 Agent 开发核心挑战：对话历史 + 工具返回 + 系统提示词 + 知识库检索结果 —— 全部挤在同一窗口里。怎么管理这个窗口，是 Agent 工程的核心问题。

四、Temperature：控制"创造力"的旋钮

Temperature 取值 0~2，决定模型输出是"保守"还是"疯狂"。

🎚️ 三种模式对比

Temperature = 0.0（保守）
"1+1=" → "2"（100%）
永远选概率最高的，结果确定、可复现

Temperature = 0.7（平衡）
"1+1=" → "2"（85%）
          "等于2"（10%）
有一点变化，但整体靠谱

Temperature = 1.5（疯狂）
"1+1=" → "2，但在某些数学体系中..."
          "这可能是一个哲学问题..."
容易胡说八道（hallucination）

实战取值建议

场景Temperature原因
代码生成0.0 ~ 0.2需要确定性，语法不能"创意"
数学计算0.0 ~ 0.11+1 永远等于 2
事实问答0.2 ~ 0.4事实不能编造
日常对话0.7自然、有变化但不离谱
创意写作0.9 ~ 1.2需要多样性和想象力
头脑风暴1.0 ~ 1.5越离谱越好，激发灵感

Top-p 和 Top-k：采样的"过滤器"

Temperature 决定"要不要随机"，Top-p 和 Top-k 决定"从哪些候选里随机"。

  • Top-k = 40：只从概率最高的 40 个 token 中选，砍掉后面的
  • Top-p = 0.9：从概率最高的开始累加，累到 ≥ 90%，只从这些里选

实际使用中两者通常配合设置，防止模型选到"明显很烂"的词。

五、推理过程：模型怎么"想"的

从输入到输出，模型内部经历了 5 步：

🔬 完整推理流程

用户输入："今天北京天气怎么样？"

Step 1 — Tokenize（分词）
  → ["今天", "北京", "天气", "怎么", "样", "？"]

Step 2 — Embedding（向量化）
  → 每个 token 转成一个高维向量（如 4096 维的数字数组）
  → "今天" → [0.23, -0.45, 0.78, ...]

Step 3 — Transformer 处理
  → 多层神经网络（如 32 层）
  → 每层包括 Self-Attention（理解词之间关系）+ FFN（特征变换）
  → 核心是 Attention 机制：每个词"关注"其他词，理解上下文

Step 4 — 逐 token 生成（Decode）
  → 输出层给出下一个 token 的概率分布
  → 采样 → "北京" → 拼回去 → 再预测 → "今天" → ...
  → 每次只生成一个 token

Step 5 — 终止
  → 生成到 <EOS>（End of Sequence）特殊 token → 停止
  → 或者达到最大长度限制 → 强制停止

两个关键阶段

阶段英文名做什么速度特点
预填充Prefill一次性处理全部输入，理解上下文计算量大，但只做一次
解码Decode逐 token 生成输出每个 token 都要过一遍全模型，串行
💡 为什么长回答慢？Prefill 只做一次，但 Decode 是串行的——生成 100 个 token 就要跑 100 次模型。这就是为什么输出越长越慢。后面第 2 周我们会实测。

✅ 自测题（含学员回答 + 标准答案）

Q1：LLM 是"知识库"还是"概率预测器"？为什么？

🙋 你的回答

概率预测器

✅ 标准答案

概率预测器。LLM 不做检索，不做存储查询。它只是在给定上文的基础上，预测下一个最可能出现的 token。训练时见过几百万次"中国的首都是北京"，所以"猜"到"北京"概率最高。证据：LLM 会"幻觉"（一本正经胡说八道）——如果是知识库，怎么会给出错误答案？

Q2：1000 个中文字大概是多少 token？为什么比英文消耗更多？

🙋 你的回答

1200 个，因为每个文字中文消耗的更多

✅ 标准答案

约 1400~2000 token。原因：① 中文信息密度高，常见英文字（如"cat"）整个词就是一个 token，而中文字符组合多，常被拆成更小的子词，一个字拆成 1.5~2 个 token；② 中文没有自然空格分隔，tokenizer 需要猜边界，切分更碎。

Q3：Context Window 不够用了会发生什么？

🙋 你的回答

1. API 报错   2. 出现幻觉   3. 理解不了意思

✅ 标准答案

三个都对，补充第 4 点："Lost in the Middle"（中间丢失）——即使没超出窗口，窗口中间的信息也容易被模型忽略。模型天然更关注开头和结尾。Agent 开发启示：重要信息放开头或结尾，别塞中间。

Q4：写代码时 Temperature 应该设多少？为什么？

🙋 你的回答

设置 0~0.2，代码时需要更加准确，越高越奔放，容易乱说，也就是猜测不准

✅ 标准答案

0.0 ~ 0.2。代码哪怕错一个字符就报错，所以 Temperature 必须低，让模型"只选最确定的"，不冒险。Temperature 低 → 概率分布更"尖" → 几乎只选概率最高的 token → 输出确定、可复现。

Q5：用自己的话描述推理过程的 5 个步骤

🙋 你的回答

1. 拆分所需 token   2. 向量化   3. 翻译   4. 转码   5. 拼接

✅ 标准答案

  1. Tokenize（分词）✅ — 把文字切成 token
  2. Embedding（向量化）✅ — 每个 token 转成高维向量
  3. Transformer 处理（不是"翻译"）— 多层 Attention 机制理解语义关系，让每个词"关注"上下文里相关的词
  4. Decode（逐 token 生成）（不是"转码"）— 一个一个 token 预测输出，串行，每次生成一个然后拼回去再预测下一个
  5. 终止（不是"拼接"）— 遇到 EOS 停止；拼接发生在 Decode 过程中，不是独立步骤

Q6：Prefill 和 Decode 有什么区别？哪个更慢？

🙋 你的回答

越长 decode 越慢，拆分的 token 越多，每次都要去循环跑模型

✅ 标准答案

Decode 更慢。Prefill（预填充）一次性并行处理全部输入 token，只做一次。Decode（解码）逐个 token 串行生成输出，每生成一个 token 跑一遍完整模型——生成 100 个 token 就要跑 100 次，所以输出越长越慢。
