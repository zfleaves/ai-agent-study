🤖 第 3 课：Agent 工作原理

概念 第 1 周 Day 3 约 35 分钟

本课目标：理解 Agent 如何从"感知"到"行动"的完整闭环，掌握 ReAct 模式，为下周写代码打下概念基础。

一、回顾：LLM 能做什么？不能做什么？

Day 2 我们学了：LLM 是一个"猜下一个词"的概率机器。它能：

  • ✅ 理解你输入的文字
  • ✅ 生成看起来合理的回答
  • ✅ 翻译、总结、润色

但它不能：

  • ❌ 查今天的天气（它不知道"今天"是哪天）
  • ❌ 发一封邮件（它没有邮箱密码）
  • ❌ 读你电脑上的文件（它没有文件系统权限）
  • ❌ 执行代码并看结果
🔑 核心矛盾

LLM 有"脑子"（推理能力），但没"手"（执行能力）。Agent = LLM + 手。

二、Agent 三要素：感知 → 思考 → 行动

Agent 的工作模式，和人类解决问题一模一样：

🧑 人类解决问题

  1. 看：读问题，收集信息
  2. 想：分析情况，决定怎么做
  3. 做：动手执行，看结果
  4. 回到第 1 步，直到问题解决

🤖 Agent 解决问题

  1. Perceive（感知）：接收用户输入 + 环境信息
  2. Think（思考）：推理分析，决定下一步
  3. Act（行动）：调用工具，执行操作
  4. 回到第 1 步，直到任务完成

三要素详解

👁️ Perceive（感知）

Agent 接收的不仅是用户输入，还包括：

  • 用户消息（User Message）
  • 工具执行结果（Tool Result / Observation）
  • 系统提示（System Prompt）
  • 对话历史（Conversation History）
  • 环境状态（日期、时间、可用工具列表等）
🧠 Think（思考）

Agent 内部做的推理：

  • "用户想要什么？"（意图理解）
  • "我需要什么信息才能回答？"（缺口分析）
  • "我应该用哪个工具？传什么参数？"（工具选择）
  • "工具返回的结果够不够？还需要再查吗？"（结果评估）
  • "任务完成了吗？可以给用户最终答案了吗？"（终止判断）
🦾 Act（行动）

Agent 可以执行的操作：

  • 调用工具函数（查天气、搜网页、读文件、运行代码）
  • 生成最终回答（输出给用户）
  • 提出追问（信息不够时反问用户）
  • 存储信息（记录到记忆系统）

三、ReAct 模式：Agent 的核心循环

ReAct = Reasoning + Acting，是 2023 年 Google 和 Princeton 提出的 Agent 范式，至今仍是 Agent 开发最主流的模式。

📜 ReAct 论文核心思想

让 LLM 交替生成 推理文字（Thought） 和 行动指令（Action）。推理帮助模型做出更好的行动；行动的结果又反馈给模型，帮助它做出更准确的推理。两者交替，形成闭环。

ReAct 循环的完整流程

┌─────────────────────────────────────────────────────┐
│                    ReAct 循环                        │
│                                                     │
│   用户输入："北京今天天气怎么样？适合出门吗？"         │
│                                                     │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐      │
│   │ PERCEIVE │───▶│  THINK   │───▶│   ACT    │      │
│   └──────────┘    └──────────┘    └──────────┘      │
│        ▲                │               │           │
│        │                │               │           │
│        │     ┌──────────┘               │           │
│        │     │         不够，继续        │           │
│        │     ▼                          ▼           │
│        │  ┌──────────────────────────────┐          │
│        └──│   Observation（观察结果）     │          │
│           │   工具返回：北京 晴 25°C      │          │
│           └──────────────────────────────┘          │
│                                                     │
│   够了吗？ → 够了 → 生成最终回答："北京今天晴天..."   │
│           → 不够 → 回到 PERCEIVE（继续循环）         │
└─────────────────────────────────────────────────────┘

一步一步拆解

  1. 用户输入："北京今天天气怎么样？适合出门吗？"
    Agent 收到一个需要外部信息的任务。
  2. Thought（推理）："用户想知道北京天气。我需要调用天气查询工具，参数是 city='北京'。"
    Agent 分析：要回答这个问题，我需要查天气——这是工具调用的缺口。
  3. Action（行动）：调用 getWeather({ city: "北京" })
    Agent 把意图转化为具体的函数调用。
  4. Observation（观察）：工具返回 { temp: 25, weather: "晴", wind: "微风" }
    工具执行完毕，结果返回给 Agent。
  5. Thought（推理）："北京今天晴天，25°C，微风。天气很好，适合出门。不需要再查其他信息了。"
    Agent 评估：信息够了，可以给最终答案。
  6. Action（行动）：生成最终回答 → "北京今天晴天，温度 25°C，微风徐徐，非常适合出门！"
    任务完成，循环结束。
💡 关键洞察：ReAct 循环不是"一轮就结束"。Agent 可能调用 多次工具，每次都用上一次的结果来推理下一步。一个复杂任务可能循环 5-10 轮。

四、Agent 循环的终止条件

Agent 什么时候停止循环？有 3 种情况：

✅ 正常终止

Agent 判断信息足够，生成最终回答给用户。

Thought: 信息够了
Action: Answer("北京今天
晴天，适合出门")

⚠️ 异常终止

超出最大步数（maxSteps），或工具连续失败，强制终止。

// 第 10 轮，工具还没返回
// Agent: "我需要再查一次..."
// 系统: maxSteps=10，强制停止
最终输出: "抱歉，我暂时无法
完成这个任务"
⚠️ 常见问题：无限循环

没有 maxSteps 限制的 Agent 可能陷入死循环——每次都认为"信息还不够"，反复调用工具。这就是为什么 maxSteps 是 Agent 的必需品，不是可选项。

五、Tool-use：Agent 的"手"

Tool-use（工具调用）是 Agent 区别于普通 chatbot 的核心能力。它让模型不只是"说"，而是能"做"。

一个工具长什么样？

📝 人类理解

函数名：查天气
输入：  城市名（字符串）
输出：  { 温度, 天气, 风力 }
描述：  查询指定城市的实时天气

🤖 模型理解（JSON Schema）

{
  "name": "getWeather",
  "description": "查询指定
城市实时天气",
  "parameters": {
    "city": {
      "type": "string",
      "description": "城市名",
      "required": true
    }
  }
}

模型不需要知道函数内部怎么实现——它只需要知道：这个工具叫什么、干什么用、需要什么参数、参数是什么类型。

💡 类比：Tool 的 JSON Schema 就像 API 文档。模型读文档 → 决定调用哪个 API → 传正确的参数 → 拿到结果 → 继续处理。你写前端调后端 API 也是这个流程！

Agent 如何选择工具？

用户："帮我查一下北京和上海的天气，哪个更热？"

Agent 推理过程：
  Thought 1: "需要查两个城市的天气。先查北京。"
  Action 1: getWeather({ city: "北京" })
  Observation 1: { temp: 25, weather: "晴" }

  Thought 2: "北京 25°C。现在查上海。"
  Action 2: getWeather({ city: "上海" })
  Observation 2: { temp: 30, weather: "多云" }

  Thought 3: "北京 25°C vs 上海 30°C。上海更热。信息够了。"
  Answer: "上海（30°C）比北京（25°C）更热，建议注意防暑。"

// 这个任务调用了 2 次工具，3 轮循环完成
✅ 前端工程师的直觉

如果你写过前端，这个模式你应该很熟悉：

  • Tool Schema = TypeScript 的 interface/type 定义
  • Tool 调用 = 调用一个 async function
  • Observation = await 函数返回的 Promise 结果
  • ReAct 循环 = 一个 while 循环，直到条件满足才 break

六、ReAct vs 其他 Agent 模式

模式 工作方式 适用场景 复杂度
ReAct 边想边做，交替进行 大多数场景，灵活应变 ⭐⭐
Plan-Execute 先制定完整计划，再逐步执行 任务明确、步骤固定 ⭐⭐⭐
Reflection 执行后自我评估，发现问题修正 代码生成、写作、需要质量保证 ⭐⭐⭐
Multi-Agent 多个 Agent 分工协作 复杂项目、多角色场景 ⭐⭐⭐⭐

我们第 1-4 周聚焦 ReAct，这是所有高级模式的基础。后面的 Plan-Execute、Reflection、Multi-Agent 都是 ReAct 的变体或扩展。

八、本课总结

✅ 今天你学到了什么
  1. Agent = LLM + 工具：LLM 有脑子没手，Agent 给它装上"手"
  2. Agent 三要素：Perceive（感知）→ Think（思考）→ Act（行动）
  3. ReAct 循环：Reasoning + Acting 交替，形成闭环，直到任务完成
  4. Tool-use：通过 JSON Schema 描述工具，模型自主选择调用
  5. 终止条件：正常终止（回答用户）vs 强制终止（maxSteps）

  │
  ▼
Agent（智能体）
  │
  │  + Memory（记忆系统）

九、自测题

  1. Agent 和普通 LLM chatbot 最大的区别是什么？
  2. ReAct 循环中的 Thought、Action、Observation 分别是什么？
  3. 为什么 Agent 需要设置 maxSteps？不设会怎样？
  4. 一个工具（Tool）需要告诉模型哪些信息？
  5. 如果 Agent 调用了工具但返回了错误，接下来会发生什么？
点击展开参考答案
  1. Agent 能调用工具"做事"（查天气、搜网页、执行代码），chatbot 只能"说"
  2. Thought：Agent 的内部推理；Action：调用工具或输出回答；Observation：工具返回的结果
  3. 防止无限循环——Agent 可能反复调用工具，永远不结束。maxSteps 是安全网
  4. 名称、功能描述、参数列表（每个参数的类型、是否必填、描述）——就像一个 API 文档
  5. Agent 会重新推理：可能换参数重试、换一个工具、或者告诉用户"我做不到"（第 4 周会详细讲错误处理）
