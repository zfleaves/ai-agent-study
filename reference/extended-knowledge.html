1. K8s（Kubernetes）— Agent 服务编排

是什么

Kubernetes 是容器编排平台。Docker 管一个容器，K8s 管一群容器（自动扩缩、负载均衡、滚动更新、服务发现）。

面试怎么聊

"我们用 Docker Compose 做开发环境，生产环境会考虑 K8s 来做自动扩缩容。Agent 服务的一个特点是请求耗时不均——简单对话 2 秒，复杂任务可能要 30 秒，K8s 的 HPA（水平自动扩缩）可以根据 GPU 利用率或请求队列长度自动调整副本数。"

关键概念

  • Pod：最小部署单元，一个 Pod 里跑一个 Agent 服务容器
  • HPA：水平自动扩缩，Agent 请求多了自动加 Pod
  • Service：负载均衡，把请求分发到多个 Agent Pod
  • ConfigMap/Secret：管理 API Key、模型配置

延伸阅读

  • K8s 官方概念概述
  • 搜索："Kubernetes 部署 LLM 服务"

2. Redis / PostgreSQL — Agent 状态持久化

📌 已纳入主课程

此话题已在 第 9 周（记忆系统 + 状态持久化）中作为核心内容。SQLite 做开发环境持久化，Redis 做生产环境会话缓存。以下内容作为补充阅读。

是什么

Agent 的对话历史、用户偏好、任务状态需要存下来。Redis 做缓存和会话（快但丢数据），PostgreSQL 做持久存储（慢但不丢）。

面试怎么聊

"Agent 的状态管理分两层：热数据用 Redis 存当前会话和工具调用中间结果，冷数据用 PostgreSQL 存用户偏好和历史任务。比如用户的对话历史缓存到 Redis（TTL 24 小时），长期偏好（"用户喜欢简洁回答"）持久化到 PostgreSQL。"

关键概念

  • Redis 会话缓存：存对话历史，TTL 过期自动清理
  • PostgreSQL JSONB：存 Agent 状态，支持灵活查询
  • pgvector 扩展：PostgreSQL 也能做向量检索，小规模场景可以不用单独的向量数据库

延伸阅读

  • Redis 客户端缓存
  • 搜索："pgvector vs Chroma 对比"

3. MT-Bench / AgentBench — 模型评估基准

是什么

MT-Bench 评估模型的多轮对话能力，AgentBench 评估模型的Agent 任务执行能力。面试官问"你怎么评估 Agent"时，提这些基准说明你有关注学术界。

面试怎么聊

"评估 Agent 分两层：离线评估用 MT-Bench 和 AgentBench 这类标准基准看模型基础能力，在线评估用我们自己的 Eval 数据集测具体业务场景的工具选择准确率和任务完成率。两个互补，离线筛模型，在线验效果。"

关键概念

  • MT-Bench：80 个多轮问题，GPT-4 当裁判打分，评估对话质量
  • AgentBench：8 个环境（操作系统、数据库、网页等），评估 Agent 的任务执行能力
  • AlpacaEval：更轻量的自动评估，适合快速对比两个模型

延伸阅读

  • MT-Bench 论文 (arXiv)
  • AgentBench 论文 (arXiv)

4. TensorRT-LLM — GPU 推理加速

是什么

NVIDIA 官方的 LLM 推理加速引擎。vLLM 是社区方案，TensorRT-LLM 是 NVIDIA 亲儿子，用 GPU 更极致但配置更复杂。

面试怎么聊

"我们的开发环境用 Ollama，生产环境对比过 vLLM 和 TensorRT-LLM。vLLM 部署简单、社区活跃，适合快速上线；TensorRT-LLM 在 NVIDIA GPU 上推理速度更快 15-30%，但需要编译模型，更新迭代成本高。我们目前选 vLLM，后续如果吞吐量成为瓶颈再切 TensorRT-LLM。"

关键概念

  • PagedAttention（vLLM）：社区方案，KV Cache 分页管理，部署简单
  • TensorRT-LLM：NVIDIA 官方，图优化 + 算子融合，极致性能
  • 选型建议：先 vLLM 快速上线，性能不够再上 TensorRT-LLM

延伸阅读

  • TensorRT-LLM GitHub
  • 搜索："vLLM vs TensorRT-LLM benchmark 2025"

5. Celery / BullMQ — Agent 异步任务队列

是什么

Agent 处理复杂任务时可能需要几十秒，不能让用户干等。Celery（Python）和 BullMQ（Node.js）都是异步任务队列：用户提交任务 → 放入队列 → 后台执行 → 完成后通知。

面试怎么聊

"Agent 的任务分同步和异步两种。简单对话同步返回（Streaming），复杂任务（如代码生成、数据分析）走异步队列。我们用 BullMQ（TS 生态）做任务调度，用户提交任务后拿到一个 taskId，前端轮询或 WebSocket 推送结果。Celery 在 Python 生态是同样的角色。"

关键概念

  • Producer/Consumer：Agent 服务是 Producer，Worker 是 Consumer
  • 任务状态：pending → running → completed/failed
  • WebSocket 推送：比轮询更优雅，任务完成主动推给前端
  • BullMQ（TS）：基于 Redis，适合 Node.js 生态
  • Celery（Python）：基于 RabbitMQ/Redis，Python 生态标配

延伸阅读

  • BullMQ 官方文档
  • Celery 官方文档

6. P-Tuning / QLoRA — LoRA 之外的微调方法

是什么

LoRA 是 PEFT（参数高效微调）的一种，还有 P-Tuning v2（在输入层加可训练 token）和 QLoRA（LoRA + 4-bit 量化，显存更省）。

面试怎么聊

"我们团队主要用 LoRA 做 Agent 的指令微调，因为它在 Tool-calling 场景效果最好。QLoRA 是 LoRA 的量化版，适合显存不够的情况。P-Tuning v2 更适合 NLU 任务但对 Tool-calling 的支持不如 LoRA。选型上，Agent 场景优先 LoRA，资源受限时用 QLoRA。"

关键概念

  • LoRA：在 Attention 层加低秩矩阵，训练参数量极小（~0.1%）
  • QLoRA：LoRA + 4-bit 量化，显存需求再降 4x，效果几乎不降
  • P-Tuning v2：在每层输入加可训练的 prompt token，适合分类/抽取类任务
  • 选型口诀：Agent 用 LoRA，显存不够用 QLoRA，NLU 用 P-Tuning

延伸阅读

  • QLoRA 论文 (arXiv)
  • P-Tuning v2 论文 (arXiv)

7. OpenAI / Anthropic API 深度使用

是什么

JD 常要求"有 OpenAI/Anthropic API 深度调用经验"。核心不是"调过 API"，而是理解 Streaming、Tool Use、System Prompt、Token 管理这些机制在不同 Provider 下的差异。

面试怎么聊

"我们项目中同时接入了 DeepSeek、OpenAI 和 Anthropic。通过 Vercel AI SDK 做 Provider 抽象，一套 Agent 代码可以切换不同模型。我理解各家 API 的差异：OpenAI 的 Tool Use 是 function_call 字段，Anthropic 是 tool_use content block，DeepSeek 兼容 OpenAI 格式。Streaming 上各家都支持 SSE，但 Anthropic 的 SSE 事件类型更丰富。"

关键差异

特性 OpenAI Anthropic DeepSeek
Tool Use 格式 function_call + tool_calls tool_use content block 兼容 OpenAI 格式
Streaming SSE（delta 增量） SSE（事件类型更丰富） 兼容 OpenAI 格式
System Prompt 顶层 system 字段 顶层 system 字段（更重视） 兼容 OpenAI 格式
上下文窗口 128K（GPT-4o） 200K（Claude 3.5） 128K（DeepSeek V4）
